# 机器码与汇编器

本章的目标，是大致把握构成计算机的组件，以及我们要制作的 C 编译器应该输出什么样的代码。这里暂时不会深入具体的 CPU 指令等细节。首先重要的是掌握概念。

## CPU 与内存

构成计算机的组件，大体可以分为 CPU 和内存。内存是能够保存数据的设备；CPU 则是在读写内存的同时执行某些处理的设备。

从概念上说，对 CPU 来说，内存看起来像一个可以随机访问的巨大字节数组。CPU 访问内存时，会用数值指定自己想访问内存的第几个字节，这个数值称为“地址”。例如，“从地址 16 读取 8 字节数据”，意思就是从这个看起来像字节数组的内存的第 16 个字节开始读取 8 字节数据。同一件事也可以说成“从 16 号地址读取 8 字节数据”。

CPU 执行的程序，以及该程序读写的数据，都存放在内存中。CPU 内部保存着“当前正在执行的指令的地址”，它会从这个地址读出指令，执行其中写明的操作，然后继续读出并执行下一条指令。这个当前正在执行的指令地址称为“程序计数器”（PC）或“指令指针”（IP）。CPU 所执行的程序形式本身称为“机器码”（machine code）。

程序计数器并不一定只会线性地前进到下一条指令。使用 CPU 的“分支指令”（branch instruction）这类指令，可以把程序计数器设置为下一条指令以外的任意地址。`if` 语句和循环等结构就是通过这一功能实现的。把程序计数器设置到下一条指令以外的位置，称为“跳转”或“分支”。

除了程序计数器之外，CPU 还拥有少量保存数据的区域。例如 Intel 和 AMD 的处理器中，有 16 个可以保存 64 位整数的区域。这种区域称为“寄存器”（register）。从 CPU 角度看，内存是外部设备，读写需要一些时间；而寄存器位于 CPU 内部，可以无延迟地访问。

许多机器码指令的形式是：使用两个寄存器的值进行某种运算，并把结果写回寄存器。因此，程序的执行可以理解为：CPU 从内存把数据读入寄存器，在寄存器之间进行某种运算，然后把结果写回内存，程序就这样向前推进。

某种特定机器码指令的总称，称为“指令集架构”（instruction set architecture, ISA）或“指令集”。指令集并不是只有一种，理论上每种 CPU 都可以自由设计。不过，如果机器码层面不兼容，同一个程序就无法运行，因此指令集的变种并没有那么多。在 PC 上，使用的是 Intel 以及兼容芯片厂商 AMD 的 x86-64 指令集。x86-64 是主要指令集之一，但并不是 x86-64 独占市场。例如 iPhone 和 Android 使用的是 ARM 指令集。

## 专栏：x86-64 指令集的名称

x86-64 有时也被称为 AMD64、Intel 64、x64 等。同一个指令集之所以有这么多名称，是有历史背景的。

x86 指令集是 Intel 在 1978 年制作的，但把它扩展到 64 位的是 AMD。2000 年前后，64 位处理器的需求逐渐出现，而 Intel 当时全公司投入到名为 Itanium 的全新指令集上，刻意没有推进会与之竞争的 64 位版 x86。AMD 抓住这个空隙，制定并公开了 64 位版 x86 的规格，这就是 x86-64。之后，可能是出于品牌策略，AMD 将 x86-64 改名为 AMD64。

后来 Itanium 的失败变得明显，Intel 除了制作 64 位版 x86 以外已经没有其他选择。但那时 AMD64 的实际芯片已经有了一定出货量，因此再去制定一个与其相似但并不相同的扩展指令集也很困难，Intel 最终也采用了兼容 AMD 的指令集。据说 Microsoft 也施加了保持兼容性的压力。当时 Intel 给与 AMD64 几乎完全相同的指令集取名为 IA-32e 并采用。它没有用“64”，而是使用 IA-32e（Intel Architecture 32 extensions）这个名字，这似乎透露出一种对未能成功的指令集的留恋：64 位 CPU 的正统仍然是 Itanium。后来 Intel 转向彻底放弃 Itanium 的方针，IA-32e 也被改名为更普通的 Intel 64。Microsoft 或许是嫌名字太长，把 x86-64 称为 x64。

由于上述原因，x86-64 拥有许多不同名称。

在开源项目中，不含特定公司名称的 x86-64 这个叫法似乎更常被偏好。本书也统一使用 x86-64 这个名称。

## 什么是汇编器

机器码是 CPU 直接读取的东西，因此只考虑了 CPU 的方便，并没有考虑对人类是否易用。用二进制编辑器编写这种机器码并非不可能，但会非常痛苦。于是汇编器被发明出来。汇编语言是一种几乎与机器码一一对应的语言，但比机器码更容易被人阅读。

如果一个编译器输出的是 native binary，而不是面向虚拟机或解释器，那么它通常会以输出汇编为目标。即使某些编译器看起来像是直接输出机器码，常见构成也是先输出汇编，然后在后台启动汇编器。本书制作的 C 编译器也会输出汇编。

把汇编代码转换为机器码，有时也称为“编译”；但为了强调输入是汇编，也会特别称为“汇编”（assemble）。

读者也许以前在某处见过汇编。如果还没有见过，现在正是一个好机会。可以使用 `objdump` 命令反汇编一个合适的可执行文件，把该文件中包含的机器码显示为汇编。下面是反汇编 `ls` 命令的结果。

```console
$ objdump -d -M intel /bin/ls
/bin/ls:     file format elf64-x86-64

Disassembly of section .init:

0000000000003d58 <_init@@Base>:
  3d58: 48 83 ec 08             sub     rsp,0x8
  3d5c: 48 8b 05 7d b9 21 00    mov     rax,QWORD PTR [rip+0x21b97d]
  3d63: 48 85 c0                test    rax,rax
  3d66: 74 02                   je      366a <_init@@Base+0x12>
  3d68: ff d0                   call    rax
  3d6a: 48 83 c4 08             add     rsp,0x8
  3d6e: c3                      ret
...
```

在作者的环境中，`ls` 命令包含大约 2 万条机器码指令，因此反汇编结果也会接近 2 万行，十分庞大。这里只列出了最开始的一小部分。

汇编基本上是每条机器码对应一行的结构。来看下面这一行。

```asm
3d58:   48 83 ec 08           sub     rsp,0x8
```

这一行是什么意思呢？`3d58` 是机器码所在内存的地址。也就是说，当 `ls` 命令执行时，这一行的指令会被放在内存的 `0x3d58` 地址处，当程序计数器为 `0x3d58` 时，这条指令就会被执行。后面跟着的 4 个十六进制数值是真正的机器码。CPU 读取这些数据，并把它作为指令执行。`sub rsp,0x8` 是与这条机器码指令对应的汇编。CPU 指令集之后会专门分章说明；这里这条指令表示从名为 RSP 的寄存器中减去 8（subtract = 减）。

## C 以及与之对应的汇编

### 简单示例

为了把握 C 编译器会生成什么样的输出，我们来比较 C 代码和与之对应的汇编代码。作为最简单的例子，考虑下面的 C 程序。

```c
int main() {
  return 42;
}
```

如果写有这个程序的文件名为 `test1.c`，可以像下面这样编译，并确认 `main` 确实返回了 42。

```console
$ cc -o test1 test1.c
$ ./test1
$ echo $?
42
```

在 C 中，`main` 函数返回的值会成为整个程序的退出码。程序的退出码不会显示在屏幕上，但会被隐式设置到 shell 的 `$?` 变量中，因此在命令结束后马上用 `echo` 显示 `$?`，就能看到该命令的退出码。这里可以看到它正确返回了 42。

那么，与这个 C 程序对应的汇编程序如下。

```asm
.intel_syntax noprefix
.globl main
main:
        mov rax, 42
        ret
```

在这段汇编中，定义了全局标签 `main`，标签之后跟着 `main` 函数的代码。这里把数值 42 设置到名为 RAX 的寄存器中，然后从 `main` 返回。可以放入整数的寄存器包括 RAX 在内一共有 16 个，但约定是函数返回时 RAX 中的值就是函数返回值，所以这里把值设置到了 RAX。

让我们实际汇编并运行这个汇编程序。汇编文件的扩展名是 `.s`，所以把上面的汇编代码写入 `test2.s`，然后执行下面的命令。

```console
$ cc -o test2 test2.s
$ ./test2
$ echo $?
42
```

和 C 的时候一样，退出码变成了 42。

粗略地说，C 编译器就是一个读取 `test1.c` 这样的 C 代码，并输出 `test2.s` 这样的汇编的程序。

### 包含函数调用的示例

作为稍微复杂一点的例子，我们来看看包含函数调用的代码会被转换成什么样的汇编。

函数调用不同于单纯的跳转，在被调用函数结束之后，必须回到原本正在执行的位置。原本正在执行的地址称为“返回地址”。如果函数调用只有一层，那么返回地址保存在 CPU 的某个合适寄存器里就可以；但函数调用可以任意加深，因此返回地址必须保存在内存中。实际上，返回地址会保存在内存上的栈中。

栈可以只用一个保存栈顶地址的变量来实现。保存这个栈顶的存储区域称为“栈指针”。为了支持使用函数的编程，x86-64 提供了专用于栈指针的寄存器，以及使用该寄存器的指令。把数据放到栈上称为“push”，取出栈上的数据称为“pop”。

接下来看看函数调用的实际例子。请考虑下面的 C 代码。

```c
int plus(int x, int y) {
  return x + y;
}

int main() {
  return plus(3, 4);
}
```

与这段 C 代码对应的汇编如下。

```asm
.intel_syntax noprefix
.globl plus, main

plus:
        add rsi, rdi
        mov rax, rsi
        ret

main:
        mov rdi, 3
        mov rsi, 4
        call plus
        ret
```

第 1 行是指定汇编语法的指令。第 2 行以 `.globl` 开头，它告诉汇编器，`plus` 和 `main` 这两个函数不是文件作用域内的函数，而是从整个程序都能看到的函数。这里暂时可以忽略它。

先关注 `main`。在 C 中，`main` 带着参数调用了 `plus`。在汇编层面，约定是第 1 个参数放入 RDI 寄存器，第 2 个参数放入 RSI 寄存器，所以 `main` 的前两行就按这个约定设置了值。

`call` 是调用函数的指令。具体来说，`call` 会执行以下操作。

- 把 `call` 下一条指令的地址（本例中是 `ret`）push 到栈上
- 跳转到作为 `call` 参数给出的地址

因此，当 `call` 指令执行时，CPU 就开始执行 `plus` 函数。

请关注 `plus` 函数。`plus` 函数有 3 条指令。

`add` 是执行加法的指令。在这个例子中，RSI 寄存器和 RDI 寄存器相加的结果会写入 RSI 寄存器。x86-64 的整数运算指令通常只接收两个寄存器，因此结果会以覆盖第 1 个参数寄存器的形式保存。

函数返回值按约定要放入 RAX。因此，我们希望把加法结果放入 RAX，所以需要把值从 RSI 复制到 RAX。这里使用 `mov` 指令完成这一点。`mov` 是 move 的缩写，但实际上它并不移动数据，只是复制数据。

在 `plus` 函数最后，调用 `ret` 从函数返回。具体来说，`ret` 会执行以下操作。

- 从栈中 pop 出一个地址
- 跳转到该地址

也就是说，`ret` 是把 `call` 所做的事情复原，并恢复调用方函数执行的指令。`call` 和 `ret` 就这样被定义为成对的指令。

从 `plus` 返回后所在的位置，是 `main` 的 `ret` 指令。在原始 C 代码中，`plus` 的返回值会直接作为 `main` 的返回值返回。这里 `plus` 的返回值已经放在 RAX 中，因此直接从 `main` 返回，就可以把它原样作为 `main` 的返回值。

## 本章小结

本章概要说明了计算机内部如何工作，以及 C 编译器应该做什么。看到汇编和机器码时，它们看起来似乎是与 C 相距甚远、乱糟糟的数据块；但也许很多读者会觉得，它们实际上意外地直接反映了 C 的结构。

本书目前几乎还没有说明具体机器码，所以你可能还不理解 `objdump` 显示出的每条汇编指令是什么意思；但应该已经能想象到，每一条指令本身并没有做什么特别复杂的事。在本章阶段，只要能获得这种感觉就足够了。

本章要点总结如下。

- CPU 通过读写内存推进程序执行
- CPU 执行的程序和程序处理的数据都在内存中，CPU 从内存依次读取机器码指令并执行
- CPU 中有称为寄存器的小型存储区域，许多机器码被定义为寄存器之间的操作
- 汇编是把机器码变得更容易被人阅读的语言，C 编译器通常输出汇编
- C 的函数在汇编中也会成为函数
- 函数调用使用栈实现

## 专栏：在线编译器

观察 C 代码及其编译结果，是学习汇编语言的好方法。不过，反复编辑源代码、编译、再确认输出的汇编，意外地麻烦。有一个非常好的网站可以减少这些步骤，那就是 Compiler Explorer（俗称 godbolt）。在 Compiler Explorer 左半边的文本框中输入代码，右半边就会实时显示与之对应的汇编输出。想确认 C 代码会被转换成什么样的汇编时，用这个网站会很方便。
