# 指针与字符串 literal

到目前为止，我们的语言已经逐渐能进行一些有意义的计算了，但它甚至还不能显示 `Hello world`。差不多该加入字符串，让程序能够输出有意义的消息了。

C 的字符串 literal 和 `char` 类型、全局变量、数组密切相关。请看下面这个函数。

```c
int main(int argc, char **argv) {
  printf("Hello, world!\n");
}
```

上面的代码会被编译成类似下面的代码。这里假定 `msg` 是一个不会与其他标识符冲突的唯一标识符。

```c
char msg[15] = "Hello, world!\n";

int main(int argc, char **argv) {
  printf(msg);
}
```

我们的编译器还缺少一些支持字符串 literal 所需的功能。为了支持字符串 literal，并能通过 `printf` 等显示消息，本章会按顺序实现下面这些功能。

1. 单目 `&` 和单目 `*`

2. 指针

3. 配列

3. 数组

4. 全局变量

5. char型

5. `char` 类型

6. 字符串 literal

此外，本章还会加入测试上述功能所需的其他功能。

## 步骤 16：单目 `&` 和单目 `*`

在这个步骤中，作为实现指针的第一步，我们实现返回地址的单目 `&`，以及引用地址的单目 `*`。

这些运算符本来是返回指针类型的值，或接收指针类型的值的运算符。但我们的编译器还没有整数以外的类型，所以先用整数类型代替指针类型。也就是说，`&x` 会把变量 `x` 的地址作为普通整数返回；而 `*x` 会把 `x` 的值视为地址，并从该地址读取值。

实现这样的运算符后，下面这样的代码就能工作。

```c
x = 3;
y = &x;
return *y; // 返回 3
```

另外，利用局部变量在内存中连续分配这一点，也可以通过指针强行间接访问栈上的变量。下面的代码假定变量 `x` 位于栈上变量 `y` 上方 8 字节的位置。

```c
x = 3;
y = 5;
z = &y + 8;
return *z; // 返回 3
```

在这种不区分指针类型和整数类型的实现中，例如 `*4` 这样的表达式会变成从地址 4 读取值的表达式。这里先暂且接受这一点。

实现相对简单。下面展示加入单目 `&` 和单目 `*` 后的语法。请按照这个语法修改 parser，把单目 `&` 和单目 `*` 分别读取为 `ND_ADDR` 和 `ND_DEREF` 类型的节点。

```ebnf
unary = "+"? primary
      | "-"? primary
      | "*" unary
      | "&" unary
```

需要加入 code generator 的修改很少。修改点如下。

```c
case ND_ADDR:
  gen_lval(node->lhs);
  return;
case ND_DEREF:
  gen(node->lhs);
  printf(" pop rax\n");
  printf(" mov rax, [rax]\n");
  printf(" push rax\n");
  return;
```

## 步骤 17：废除隐式变量定义，引入 `int` 关键字

到目前为止，变量和函数返回值都被隐式地视为 `int`。因此我们没有特意像 `int x;` 这样把变量和类型名一起定义，而是把所有新标识符都视为新变量名。今后不能再采用这种假设，所以先改造这一点。请实现下面这些功能。

- 不再把新标识符自动视为变量名；如果出现未定义变量，就报错。

- 让变量通过 `int x;` 这种形式定义。不需要支持 `int x = 3;` 这样的初始化式，也不需要支持 `int x, y;`。只实现尽可能简单的形式。

- 函数以前写成 `foo(x, y)` 这样的形式，现在改造为 `int foo(int x, int y)` 这样的形式。目前 top level 应该只有函数定义，所以 parser 先读取 `int`，之后必然是函数名，再之后读取 `int <参数名>` 这样的序列即可。不需要支持更复杂的语法，也不需要为了“未来扩展”做额外准备。只要写出足以读取 `int <函数名>(<由 int <变量名> 重复组成的参数列表>)` 的代码即可。

## 步骤 18：引入指针类型

### 定义表示指针的类型

在这个步骤中，以前类型名只允许 `int`，现在改为允许 `int` 后面跟 0 个以上的 `*` 作为类型名。也就是说，要能够解析 `int *x` 或 `int ***x` 这样的定义。

“指向 `int` 的指针”这样的类型当然需要能在编译器中处理。例如，如果变量 `x` 是指向 `int` 的指针，那么编译器必须知道表达式 `*x` 是 `int` 类型。类型可以任意复杂，例如“指向指向指向 `int` 的指针的指针”，所以不能只用固定大小的类型来表示。

那么该怎么做呢？使用指针。到目前为止，通过 map 关联到变量的信息只有变量相对于栈上 base pointer（`RBP`）的 offset。请修改这一点，让变量也能持有类型。粗略地说，变量类型应该是下面这样的结构体。

```c
struct Type {
  enum { INT, PTR } ty;
  struct Type *ptr_to;
};
```

这里 `ty` 可以取两个值之一：`int` 类型，或“指向某类型的指针”类型。`ptr_to` 只有在 `ty` 是“指向某类型的指针”时才有意义，此时里面保存指向“某类型”对应 `Type` 对象的指针。例如“指向 `int` 的指针”，其类型数据结构在内部如下。

```text
Pointer --ptr_to--> int
```

如果是“指向指向 `int` 的指针的指针”，则如下。

```text
Pointer --ptr_to--> Pointer --ptr_to--> int
```

这样一来，编译器内部就可以表示任意复杂的类型。

### 向指针所指的值赋值

赋值表达式的左边不是简单变量名时，例如 `*p=3` 这样的表达式，该如何编译呢？这类表达式与左边是简单变量时的基本概念并没有变化。在这个例子中，只要把 `*p` 作为左值编译，使它生成 `p` 的地址即可。

编译表示 `*p=3` 的语法树时，会递归下降语法树并生成代码，而最先被调用的是把 `*p` 作为左值编译的 code generator。

这个 code generator 会根据给定语法树的类型分支处理。对于简单变量，如前所述，会输出用于生成该变量地址的代码。但这里给出的是 dereference 运算符，所以需要采用不同动作。如果给出的是 dereference 运算符，请把其中的语法树作为“右值”编译。这样它应该会被编译成某种计算地址的代码（否则就无法 dereference 该结果）。然后只要把这个地址原样留在栈上即可。

完成到这个阶段后，应该就能编译下面这样的语句。

```c
int x;
int *y;
y = &x;
*y = 3;
return x; // -> 3
```

## 步骤 19：实现指针的加法和减法

在这个步骤中，让指针类型的值 `p` 可以写成 `p+1` 或 `p-5` 这样的表达式。这看起来和普通整数加法一样，但实际上是相当不同的运算。`p+1` 不是给 `p` 保存的地址加 1，而是让它成为指向 `p` 下一个元素的指针，所以必须给 `p` 加上指针所指数据类型的宽度。例如，如果 `p` 指向 `int`，在我们的 ABI 中，`p+1` 以地址字节数来看就是加 4。另一方面，如果 `p` 是指向指向 `int` 的指针的指针，`p+1` 就是加 8。

因此在指针加减法中，需要一种知道类型大小的方法。但目前 `int` 是 4，指针是 8，所以先按这种固定规则写代码即可。

在这个阶段还没有连续分配内存的方法（我们的编译器还没有数组），所以写测试稍微有些困难。这里可以借助外部编译器，让它那边执行 `malloc`，而在自己编译器的输出中使用那个 helper 函数来写测试。可以像下面这样测试。

```c
int *p;
alloc4(&p, 1, 2, 4, 8);
int *q;
q = p + 2;
*q; // -> 4
q = p + 3;
return *q; // -> 8
```

### 专栏：`int` 和 `long` 的大小

像 x86-64 System V ABI 这样，`int` 为 32 bit，`long` 和指针为 64 bit 的数据模型称为 LP64。它的意思是 `long` 和 pointer 为 64 bit。同样是在 x86-64 上的 ABI，Windows 采用的是 LLP64，也就是 `int` 和 `long` 为 32 bit，`long long` 和指针为 64 bit 的数据模型。

LP64 和 LLP64 中 `long` 的大小不同，所以 ABI 不兼容。例如，如果创建一个包含 `long` 成员的结构体，把整个结构体原样写入文件，读取时再把文件上的数据直接 cast 成该结构体处理，那么 Unix 和 Windows 之间就无法互相传递并读取这个文件。

C 规范规定，`int` 是“该机器上自然的整数大小”（A "plain" int object has the natural size suggested by the architecture of the execution environment）。这样说的话，似乎 64 bit 机器上就必须让 `int` 成为 64 bit；但什么是“自然”是主观问题，而且即使在 64 bit 机器上，通常也可以自然地处理 32 bit 运算，所以在 64 bit 机器上让 `int` 为 32 bit 并不一定错误。现实地考虑，如果把 `int` 设为 64 bit，会产生下面这些问题。

- 需要 `int` 拥有 64 bit 大小的情况很少，所以把 `int` 设为 64 bit 只会浪费内存。

- 如果 `short` 是 16 bit，而 `int` 和 `long` 都是 64 bit，就会没有表示 32 bit 整数的类型。

基于上述理由，现存的大多数 64 bit 机器中 `int` 都是 32 bit。不过也确实存在 `int` 为 64 bit 的 ILP64。例如，据说早期 Cray 超级计算机就是 ILP64。

## 步骤 20：`sizeof` 运算符

`sizeof` 看起来像函数，但在语法上是单目运算符。C 中大多数运算符都是符号，但在语法上并没有理由要求运算符必须是符号，`sizeof` 实际上就是例外。

稍微复习一下 `sizeof` 运算符的行为。`sizeof` 是返回参数表达式的类型在内存中占多少字节的运算符。例如在我们的 ABI 中，`sizeof(x)` 在 `x` 是 `int` 时返回 4，在 `x` 是指针时返回 8。`sizeof` 的参数可以写任意表达式，例如 `sizeof(x+3)` 会根据 `x+3` 这个表达式整体的类型是 `int` 还是指针，返回 4 或 8。

我们的编译器还没有数组，但如果 `x` 是数组，`sizeof(x)` 会以字节数返回整个 `x` 的大小。例如，如果 `x` 定义为 `int x[10]`，`sizeof(x)` 返回 40。如果 `x` 定义为 `int x[5][10]`，`sizeof(x)` 返回 200，`sizeof(x[0])` 返回 40，`sizeof(x[0][0])` 返回 4。

`sizeof` 运算符的参数只是为了知道类型而写的，并不是实际执行的表达式。例如写 `sizeof(x[3])` 时，实际上不会发生对 `x[3]` 的访问。表达式 `x[3]` 整体是什么类型，在编译时就可以知道，所以 `sizeof(x[3])` 会在编译时替换为该类型的大小。因此，传给 `sizeof` 的具体表达式，例如 `x[3]`，在运行时已经不存在。

下面展示 `sizeof` 的行为。

```c
int x;
int *y;

sizeof(x); // 4
sizeof(y); // 8

sizeof(x + 3); // 4
sizeof(y + 3); // 8
sizeof(*y);    // 4

// 传给 sizeof 的表达式可以是任意内容
sizeof(1); // 4

// sizeof 的结果当前是 int 类型，所以与 sizeof(int) 相同
sizeof(sizeof(1)); // 4
```

接下来实现这个 `sizeof` 运算符。为了实现 `sizeof` 运算符，需要同时修改 tokenizer 和 parser。

首先修改 tokenizer，让它把 `sizeof` 这个关键字识别为 `TK_SIZEOF` 类型的 token。

接下来修改 parser，把 `sizeof` 替换成 `int` 类型的常量。下面展示加入 `sizeof` 运算符后的语法。在下面的语法中，`sizeof` 是单目运算符，优先级和单目加号、单目减号相同。这与 C 的语法一致。

```ebnf
unary = "sizeof" unary
      | ("+" | "-")? primary
```

在这个语法中，不仅 `sizeof(x)`，像 `sizeof x` 这样的写法在语法上也是允许的，实际 C 也是如此。

在 parser 中，如果出现 `sizeof` 运算符，就像平常一样解析作为其参数的表达式，然后根据结果语法树关联的类型，把它替换成常量：如果类型是 `int` 就替换为 4，如果是指针就替换为 8。因为已经在 parser 中替换成常量，所以不需要修改 code generator。

## 步骤 21：实现数组

### 定义数组类型

这个步骤实现数组。到这个阶段为止，我们只处理过能放进寄存器大小的数据；这一次第一次出现比寄存器更大的数据。

不过，C 的语法对数组是克制的。不能把数组作为函数参数传递，也不能把数组作为函数返回值返回。如果写出意图如此的代码，被值传递的并不是数组本身，而是会自动生成指向该数组的指针并传递。也不支持把数组直接赋值给数组来复制（必须使用 `memcpy`）。

因此，不需要在函数或变量之间传递不能放进寄存器的数据。只要具备在栈上分配大于 1 word 的内存区域的功能即可。

请让编译器能够读取下面这样的变量定义。

```c
int a[10];
```

上面 `a` 的类型是数组，该数组长度为 10，元素类型是 `int`。和指针类型一样，数组类型也可以任意复杂，所以和前面的步骤一样，用 `ptr_to` 指向数组元素的类型。表示类型的结构体应该如下。

```c
struct Type {
  enum { INT, PTR, ARRAY } ty;
  struct Type *ptr_to;
  size_t array_size;
};
```

这里 `array_size` 是只有在数组类型时才有意义的字段，用来保存数组元素数量。

做到这里后，为数组在栈上分配区域应该就很简单了。要求出数组的字节大小，只需要把数组元素的字节大小乘以数组元素个数。到目前为止，所有变量应该都按 1 word 分配栈区域；请修改这一点，让数组分配数组所需的大小。

### 实现数组到指针的隐式类型转换

数组和指针经常组合使用，所以在 C 的语法上，即使不太区分指针和数组，也会在某种程度上“能工作”。但这也造成反效果，让程序员难以理解数组和指针的关系到底是什么。因此这里说明数组和指针的关系。

首先，在 C 中，数组和指针是完全不同的类型。

指针是（在 x86-64 上）8 字节值的类型。就像 `int` 定义了 `+`、`-` 等运算符一样，指针也以稍有不同的形式定义了 `+`、`-`。除此之外，指针还定义了单目 `*` 运算符，可以用它引用指针所指的位置。除了单目 `*` 之外，指针并没有特别特殊。可以说，指针是像 `int` 一样的普通类型。

另一方面，数组是大小可以为任意字节数的类型。与指针不同，数组几乎没有定义运算符。已定义的运算符只有返回数组大小的 `sizeof` 运算符，以及返回数组首元素指针的 `&` 运算符。除此之外，没有可以应用于数组的运算符。

那么为什么 `a[3]` 这样的表达式能够编译呢？在 C 中，`a[3]` 被定义为等价于 `*(a+3)`。可是数组不是没有定义 `+` 运算符吗？

这里起作用的是“数组会隐式转换为指针”这条语法。除了作为 `sizeof` 或单目 `&` 的 operand 使用时之外，数组会被隐式转换为指向该数组首元素的指针。因此，`*(a+3)` 这个表达式的含义就是：对“指向数组 `a` 首元素的指针加 3”之后得到的指针进行 dereference，结果就等同于访问数组的第 3 个元素。

在 C 中，并不存在专门用于数组访问的 `[]` 运算符。C 的 `[]` 只是通过指针访问数组元素的便捷记法。

同样，把数组作为函数参数传递时会变成指向该数组首元素的指针，或者看起来好像可以把数组直接赋给指针，这些也都是基于上述原因。

因此，编译器在实现大多数运算符时，都必须执行数组到指针的类型转换。这实现起来并不太难。除了实现 `sizeof` 和单目 `&` 的情况之外，在解析运算符 operand 后，如果它的类型是 `T` 的数组，就把它视为指向 `T` 的指针即可。在 code generator 中，数组类型的值应该生成“把该值的地址 push 到栈上”的代码。

完成到这里后，下面这样的代码应该就能工作。

```c
int a[2];
*a = 1;
*(a + 1) = 2;
int *p;
p = a;
return *p + *(p + 1)   // -> 3
```

### 专栏：语言律师

对于非常理解正式语言规范的人，有时会把语言规范比作法律，称其为“语言律师”（language lawyer）。程序员俚语词典《Jargon File》中对语言律师的解释如下。

熟练的语言律师似乎常常受到其他程序员敬重。作者在 Google 的 C++ 编译器团队工作时，团队里有一位可以说是终极语言律师的人。当大家遇到不懂的 C++ 问题时，经常会得出“问问他吧”的结论（即使是写 C++ 编译器的人，也有很多 C++ 规范中不明白的地方）。实际上，他实现过主流 C++ 编译器 Clang 的主要部分，也是 C++ 规范书的首席作者，是世界上最了解 C++ 的人之一。但即便是他也说过：“C++ 一旦觉得自己懂了，就会变得不太懂。”作者因此记得，C++ 语言规范的庞大和细节复杂度相当惊人。

本书在编译器完成度提高之前，会有意避免深入 C 语言规范的细节。这是有原因的。实现一种有规范的编程语言时，在一定程度上确实需要成为语言律师；但从一开始就过分在意细节，并不是理想的开发方法。就像画画时不应该只细致描绘某一个地方，而应先完成整体粗略草图一样，实现编程语言时，最初也需要保持平衡，不要过早进行太多语言律师式讨论。

## 步骤 22：实现数组下标

在 C 中，`x[y]` 被定义为等价于 `*(x+y)`。因此，下标的实现相对简单。只要在 parser 中把 `x[y]` 改读为 `*(x+y)` 即可。例如 `a[3]` 会变成 `*(a+3)`。

在这个语法中，`3[a]` 会展开为 `*(3+a)`，所以如果 `a[3]` 能工作，`3[a]` 也应该能工作。令人意外的是，在 C 中 `3[a]` 这样的表达式确实合法。可以试试看。

## 步骤 23：实现全局变量

差不多该让程序中可以写字符串 literal 了。在 C 中，字符串 literal 是 `char` 数组。数组已经实现了，这一点没问题；但不同之处在于，字符串 literal 不是存在于栈上的值。字符串 literal 存在于内存中的固定位置，而不是栈上。因此，为了实现字符串 literal，先加入全局变量。

到目前为止，top level 应该只允许函数定义。现在修改语法，让 top level 可以写全局变量。

变量定义和函数定义看起来很相似，因此语法分析稍微有些 tricky。请比较下面 4 个定义。

```c
int *foo;
int foo[10];
int *foo() {}
int foo() {}
```

上面两个 `foo` 是变量定义，下面两个是函数定义。但在到达会成为函数名或变量名的标识符，并读取其下一个 token 之前，无法区分二者。因此，需要先调用“读取类型名前半部分”的函数，然后读取应该出现的标识符，再向前看一个 token。如果向前看的 token 是 `"("`，就说明正在读取函数定义；否则就是在读取变量定义。

请把解析出的全局变量名放入 map，使其可以按名字查找。只有当变量名无法解析为局部变量时，才尝试把它解析为全局变量。这样就能自然实现局部变量遮蔽同名全局变量的行为。

在 parser 中，局部变量引用和全局变量引用会转换为抽象语法树中的不同节点。因为在解析阶段就能解析名字，所以在这个阶段也可以把类型区分开。

到目前为止，所有变量都应该在栈上，所以变量读写都是相对 `RBP`（base pointer）进行的。全局变量不是栈上的值，而是位于内存固定位置的值，所以要编译成直接访问其地址。请参考实际 GCC 的输出。

实际实现后，应该会惊讶于局部变量和全局变量其实相当不同。表面上可以不加区分地书写，是因为 C 语言做了很好的抽象。局部变量和全局变量在内部实际上以相当不同的方式实现。

## 步骤 24：实现字符类型

数组是可能大于 1 word 的类型，而字符是小于 1 word 的类型。到这个步骤为止，你应该已经需要编写一个函数：接收表示类型的对象，并返回该类型的字节大小。首先加入字符这个类型，然后修改该函数，让它对字符类型返回 1。

这个步骤不需要实现字符 literal（用单引号包围的字符）。请克制一次性实现更多内容的冲动，尽量保持修改很小。

因此在这个步骤中，字符真的只是一个小整数类型。写成 `movsx ecx, BYTE PTR [rax]` 时，可以从 `RAX` 指向的地址读取 1 字节并放入 `ECX`。如果不需要符号扩展，请使用 `movzx ecx, BYTE PTR [rax]` 这样的 `movzx` 指令。写入时，则像 `mov [rax], cl` 这样，使用 8 bit 寄存器作为源寄存器。

请参考实际编译器的输出。

实现这个步骤后，下面这样的代码应该就能工作。

```c
char x[3];
x[0] = -1;
x[1] = 2;
int y;
y = 4;
return x[0] + y;   // -> 3
```

### 专栏：8 bit 寄存器和 32 bit 寄存器的区别

为什么读取 1 字节值时需要使用 `movsx` 或 `movzx` 呢？读取 4 字节值时，只需要用普通的 `mov` 读入 `EAX` 这样的低 32 bit alias register，所以读取 `char` 时看起来只要用普通 `mov` load 到 `AL` 就可以了。但这样不能正确工作。这个谜题的答案在 x86-64 的规格中。

在 x86-64 中，读入低 32 bit alias register 时，高 32 bit 会被重置为 0。但是，读入低 8 bit alias register 时，高 56 bit 会保持之前的值。这是不一致的规格，但 x86-64 是历史很长的指令集，所以这种不一致存在于很多地方。

x86-64 是从 8086 这个 16 bit 处理器演进到 32 bit、64 bit 的，所以先有 `AL`，后来有 `EAX`，再后来有 `RAX`。也就是说，最初就存在“load 到 `AL` 时，`EAX` 的高 24 bit 不会 reset，而是保持原样”的规格；将其扩展到 64 bit 时，又制定了“load 到 `EAX` 时，`RAX` 的高 32 bit 会 reset”的规格。之所以制定这种破坏一致性的规格，是有明确理由的。

现代处理器会查看指令依赖关系，把无关指令（不使用前一条指令结果的指令）不断并行执行。假设指令集规定高 32 bit 不会 reset，即使高 32 bit 只是垃圾，程序也一直忽略它并只使用低 32 bit，那么最终在“生成被忽略的高 32 bit 的指令”和后续“使用同一寄存器的指令”之间，会产生虚假的依赖关系。把规格定为高 32 bit 被符号扩展并 reset 后，之前的值会被完全覆盖，因此可以切断依赖关系。因此，在把 x86 64 bit 化时，明知会损害一致性，仍然选择了更容易高速化的规格。

## 步骤 25：实现字符串 literal

这个步骤让编译器能够解析并编译用双引号包围的字符串。数组、全局变量、字符类型这些必要部件已经齐备，所以实现应该相对简单。

首先修改 tokenizer，让它在发现双引号时，读取到下一个双引号为止，并创建字符串 token。这个步骤不需要实现反斜杠转义等功能。按步骤前进很重要，即使看起来很容易实现，也请先不要做。

表示字符串 literal 数据的汇编代码，不能在生成 CPU 执行的机器码途中输出。在输出的汇编中，全局数据和代码需要分开书写，不能混在一起。也就是说，输出代码时，希望先输出代码中出现过的所有字符串 literal；但为了做到这一点去遍历语法树会很麻烦。简单做法是准备一个 vector，保存到目前为止见过的所有字符串 literal，parser 每看到一个字符串就把它加入其中。

请参考实际编译器的输出。

到这里，应该已经可以用 `printf` 输出字符串了。这是一个好机会，可以用自己制作的编程语言写一些比测试代码更复杂的程序，而不是只写显而易见的测试。例如，应该可以用自制语言写一个八皇后问题求解器。人类从发明数字计算机，到开发出能以这种程度轻松编写代码的编程语言，花了几十年。你能在几周内实现到这里，是人类以及你自己的巨大进步。

调用接收可变长参数的函数时，约定需要把浮点数参数的个数放入 `AL`。我们的编译器还没有浮点数。因此，在调用函数前总是把 `AL` 设置为 0 即可。

## 步骤 26：从文件读取输入

到目前为止，我们一直把 C 代码直接作为参数字符串传入，但输入逐渐变长了，所以差不多该像普通 C 编译器一样，改造成从命令行参数接收文件名。打开给定文件、读取其内容，并返回以 `'\0'` 结尾的字符串的函数，可以像下面这样简洁地编写。

```c
#include <errno.h>
#include <stdio.h>
#include <string.h>

// 返回指定文件的内容
char *read_file(char *path) {
  // 打开文件
  FILE *fp = fopen(path, "r");
  if (!fp)
    error("cannot open %s: %s", path, strerror(errno));

  // 检查文件长度
  if (fseek(fp, 0, SEEK_END) == -1)
    error("%s: fseek: %s", path, strerror(errno));
  size_t size = ftell(fp);
  if (fseek(fp, 0, SEEK_SET) == -1)
    error("%s: fseek: %s", path, strerror(errno));

  // 读取文件内容
  char *buf = calloc(1, size + 2);
  fread(buf, size, 1, fp);

  // 确保文件总是以 "\n\0" 结尾
  if (size == 0 || buf[size - 1] != '\n')
    buf[size++] = '\n';
  buf[size] = '\0';
  fclose(fp);
  return buf;
}
```

出于编译器实现上的方便，所有行都以换行字符结束，比“以换行字符或 EOF 结束”的数据更容易处理，所以如果文件最后一个字节不是 `\n`，这里会自动追加 `\n`。

严格来说，如果传入不能随机访问的特殊文件，这个函数无法正常工作。例如把表示标准输入的设备文件 `/dev/stdin` 或 named pipe 指定为文件名时，应该会看到 `/dev/stdin: fseek: Illegal seek` 这样的错误消息。不过实际使用中，这个函数应该没有问题。请用这个函数读取文件内容，并修改代码，把它作为输入处理。

输入文件通常包含多行，所以也增强一下显示错误消息的函数。发生错误时，如果显示输入文件名、错误所在行号以及该行内容，错误消息就会如下。

```text
foo.c:10: x = y + + 5;
                  ^ 不是表达式
```

显示这种错误消息的函数如下。

```c
// 输入文件名
char *filename;

// 用于报告错误发生位置的函数
// 用如下格式显示错误消息
//
// foo.c:10: x = y + + 5;
//                   ^ 不是表达式
void error_at(char *loc, char *msg) {
  // 取得 loc 所在行的起点和终点
  char *line = loc;
  while (user_input < line && line[-1] != '\n')
    line--;

  char *end = loc;
  while (*end != '\n')
    end++;

  // 检查找到的行是整体中的第几行
  int line_num = 1;
  for (char *p = user_input; p < line; p++)
    if (*p == '\n')
      line_num++;

  // 把找到的行连同文件名和行号一起显示
  int indent = fprintf(stderr, "%s:%d: ", filename, line_num);
  fprintf(stderr, "%.*s\n", (int)(end - line), line);

  // 用 "^" 指出错误位置并显示错误消息
  int pos = loc - line + indent;
  fprintf(stderr, "%*s", pos, ""); // 输出 pos 个空格
  fprintf(stderr, "^ %s\n", msg);
  exit(1);
}
```

这个错误消息输出例程构造相当简单，但相对而言，输出格式看起来已经比较正式。

### 专栏：错误恢复

当输入代码在语法上有错误时，许多编译器会适当地跳过出错位置，并尝试继续之后的 parse。这样做的目的是尽可能找出多个错误，而不是只找出一个。从 parser 的错误中恢复并继续 parse 的功能称为“错误恢复”。

错误恢复在早期编译器中是非常重要的功能。在 1960 年代和 1970 年代，程序员通过 time-sharing 使用计算中心的大型计算机，从带着想编译的代码过去，到得到编译结果，有时需要等一整晚。在这样的环境中，尽可能多地指出可指出的错误，是编译器的重要工作之一。早期编写的编译器教材中，错误恢复是语法分析的主要主题之一。

现在使用编译器进行开发更加交互式，所以错误恢复不再是那么重要的主题。我们开发的编译器只显示第一个错误消息。现代情况下，很多时候这样已经足够。

## 步骤 27：行注释和 block 注释

我们的编译器也逐渐演化，已经能写更正式的代码了。发展到这里，就会想要注释。本节实现注释。

C 有两种注释。一种称为行注释，从 `//` 到行末都是注释。另一种称为 block 注释，`/*` 是开始符号，`*/` 是结束符号。block 注释中的字符，除了 `*/` 这两个字符的序列之外，全部会被跳过。

语法上，注释的处理等同于一个空白字符。因此，在 tokenizer 中像跳过空白字符一样跳过注释是自然的。下面展示跳过注释的代码。

```c
void tokenize() {
  char *p = user_input;

  while (*p) {
    // 跳过空白字符
    if (isspace(*p)) {
      p++;
      continue;
    }

    // 跳过行注释
    if (strncmp(p, "//", 2) == 0) {
      p += 2;
      while (*p != '\n')
        p++;
      continue;
    }

    // 跳过 block 注释
    if (strncmp(p, "/*", 2) == 0) {
      char *q = strstr(p + 2, "*/");
      if (!q)
        error_at(p, "注释没有闭合");
      p = q + 2;
      continue;
    }
    ...
  }
}
```

这里为了寻找 block 注释的结尾，使用了 C 标准库中的 `strstr` 函数。`strstr` 会从字符串中查找字符串；如果找到传入的字符串，就返回指向其开头的指针；如果找不到，就返回 `NULL`。

### 专栏：行注释

原始 C 中只有 block 注释，行注释是在 C 被开发出来将近 30 年后的 1999 年才正式加入规范的。最初的设想中，这应该是不破坏兼容性的修改；但实际上在一些微妙情况下，原本能工作的代码会变成另一种含义。

具体来说，下面的代码如果只支持 block 注释，会被读作 `a/b`；如果支持行注释，则会被读作 `a`。

```c
a//*
// */ b
```

### 专栏：block 注释与嵌套

block 注释不能嵌套。`/*` 在注释中没有特殊含义，所以如果试图注释掉已有 block 注释：

```c
/* /* ... */ */
```

写成这样时，注释会在第一个 `*/` 结束，并在第二个 `*/` 处发生语法错误。

如果想一次性注释掉可能包含 block 注释的多行，可以使用 C preprocessor：

```c
#if 0
...
#endif
```

也就是用 `#if 0` 包起来。
