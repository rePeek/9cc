# 程序的执行映像与初始化式

到目前为止，我们的编译器已经支持了函数、全局变量、局部变量这些编程中的主要元素。读者也已经通过学习分割编译和链接，理解了如何把程序分成小块编译，并在最后把它们合并为一个文件。

本章会说明 OS 如何执行可执行文件。读完本章后，就能理解可执行文件中包含哪些数据，以及在 `main` 函数被调用之前发生了什么。

此外，本章还会说明变量初始化式，也就是下面这样的代码会如何被编译，并给我们的编译器加入对初始化式的支持。

```c
int x = 3;
int y[3] = {1, 2, 3};
char *msg1 = "foo";
char msg2[] = "bar";
```

为了支持初始化式，可能有些意外，但必须理解程序到达 `main` 之前是如何运行的。

本章说明的是一种简单执行形式：所有代码和数据都汇总在一个可执行文件中。这种执行形式的文件称为“静态链接”（static link）的可执行文件。与静态链接相对，还有一种广泛使用的执行形式叫“动态链接”（dynamic link）：一个程序的片段分散在多个文件中，在运行时在内存中结合后执行。动态链接会另开一章说明。这里先扎实理解作为基本模型的静态链接。

## 可执行文件的结构

可执行文件由 file header 和一个以上称为“segment”的区域组成。一个可执行文件通常至少有两个 segment，可执行代码和数据分别放在不同 segment 中。包含可执行代码的 segment 称为“text segment”，包含其他数据的 segment 称为“data segment”。实际可执行文件中还包含其他 segment，但为了理解机制并不需要，这里省略。

关于术语需要注意：这里的 text 和 text file 中的 text 是同一个词，但含义不同。传统上在低层领域中，表示机器码的数据称为“text”。另外，机器码本身也不过是字节序列，所以 text 也是一种数据；但说“text 和 data”时，“data”通常指“text 以外的数据”。本章中说 data 时，也表示 text 以外的数据。

作为 linker 输入的 object file 中，text 和 data 是分开放置的。linker 会把从多个 object file 读取的 text 连接起来，放入一个 text segment；同样，也会把从多个 object file 读取的数据连接起来，放入一个 data segment。

可执行文件的 file header 中，针对每个 segment 写着运行时应该放置到哪个内存地址。执行可执行文件时，OS 中称为“program loader”，或简称“loader”的程序，会根据这些信息，把 text 和 data 从可执行文件复制到内存中。

下面展示可执行文件，以及 loader 把该可执行文件读入内存后的状态。

```text
可执行文件:

  文件头
  data segment
  text segment

内存映像:

  0x7fff_ffff_ffff_ffff
  栈区域

  ...

  0x0000_0000_0005_0000
  data segment

  ...

  0x0000_0000_0004_1000
  text segment

  0x0000_0000_0000_0000
```

在这张图中的可执行文件里，假定 file header 写着这样的信息：text segment 载入到 `0x41000`，data segment 载入到 `0x50000`。

file header 中还包含应该从哪个地址开始执行的信息。例如，如果写着应该从 `0x41040` 开始执行，loader 会先像上图那样把可执行文件载入内存，然后把 stack pointer 设置为 `0x7fff_ffff_ffff_ffff`，再跳转到 `0x41040`，开始执行用户程序。

## data segment 的内容

text segment 的内容显然是机器码，那么 data segment 中放的是什么呢？答案是：data segment 中放着全局变量、字符串 literal 等。

局部变量既不直接放在 text segment 中，也不直接放在 data segment 中。局部变量是程序动态地在栈区域中创建出来的，所以刚把可执行文件载入内存时，它们并不存在。

在 C 的执行模型中，程序几乎只要把可执行文件原样读入内存，就能开始执行 `main` 函数。因此，全局变量必须做到：只要从可执行文件的 data segment 复制到内存，就已经设置好了合适的初始值。

基于这个限制，C 中不能对全局变量使用下面这种包含函数调用的初始化式。

```c
int foo = bar();
```

如果存在上面这样需要动态初始化的全局变量，那么在执行 `main` 函数之前，就必须有人执行上面的表达式。但 C 中不存在在 `main` 之前启动的初始化机制，所以无法进行这种初始化。

也就是说，全局变量的值必须在链接时完成，并能作为原样字节序列放入可执行文件。这类值只限于下面这些表达式。

- 定数式

- 常量表达式

- 全局变量或函数的地址

- 全局变量或函数的地址加上常量

像数字 literal 或字符串这样的常量表达式，显然可以直接作为固定值设置到 segment 中。

全局变量或函数的地址通常在编译时还不能确定，但 linker 完成可执行文件时通常可以确定。因此，用另一个全局变量的地址初始化指针类型全局变量，例如 `int *x = &y;` 这样的定义是合法的。linker 自己决定程序 segment 的布局，当然也知道函数和全局变量会被加载到什么地址，所以能在链接时填入 `x` 的内容。

另外，给标签地址加上常量是 linker 支持的功能，所以 `int *x = &y + 3;` 这样的定义也是合法的。

不能在初始化式中写上述模式以外的表达式。例如，不能在初始化式中使用全局变量的值（不是地址）。像 `ptrdiff_t x = &y - &z;` 这样计算两个全局变量地址差的表达式，原则上是可以在链接时求值的表达式，但 linker 不支持计算两个标签差的操作，所以这种表达式不能写在初始化式中。全局变量初始化式只允许上述有限模式。

能作为全局变量初始化式书写的表达式，用 C 表示如下。

```c
int a = 3;
char b[] = "foobar";
int *c = &a;
char *d = b + 3;
```

与各个表达式对应的汇编如下。

```asm
a:
  .long 3
b:
  .byte 0x66 // 'f'
  .byte 0x6f // 'o'
  .byte 0x6f // 'o'
  .byte 0x62 // 'b'
  .byte 0x61 // 'a'
  .byte 0x72 // 'r'
  .byte 0    // '\0'
c:
  .quad a
d:
  .quad b + 3
```

连续的 `.byte` 也可以使用 `.ascii` 记法写成 `.ascii "foobar\0"`。

### 专栏：全局变量的动态初始化

在 C 中，全局变量的内容必须静态确定；但在 C++ 中，可以用任意表达式初始化全局变量。也就是说，在 C++ 中，全局变量初始化式会在 `main` 函数被调用之前执行。它通过下面这样的机制工作。

- C++ 编译器把全局变量初始化式汇总成函数，并把该函数指针输出到名为 `.init_array` 的特殊 section 中。

- linker 把多个输入文件的 `.init_array` section 连接起来，输出到 `.init_array` segment 中（因此 `.init_array` segment 中包含函数指针数组）。

- loader 在把控制权交给 `main` 之前，先按顺序执行 `.init_array` segment 中的函数指针。

像这样，通过编译器、linker 和 program loader 的协作，全局变量的动态初始化成为可能。

如果使用和 C++ 相同的机制，C 也可以支持全局变量动态初始化，但这种功能被有意排除在 C 语言规范之外。

C 语言规范的这种设计选择，对编写程序的一方来说限制更多；但对执行程序的一方来说，即使在功能贫弱的 loader 或没有 loader 的环境中（例如计算机启动时从 ROM 直接执行的代码），也可以完全满足语言规范。因此这是取舍问题，不是哪一方更优秀的问题。

## 初始化式的语法

初始化式乍看像普通赋值表达式，但实际上初始化式和赋值表达式在语法上相当不同，存在一些只有初始化式才允许的特殊写法。这里先准确把握这些特殊写法。

首先，初始化式可以初始化数组。例如下面的表达式会把 `x` 初始化为 `x[0]`、`x[1]`、`x[2]` 分别为 0、1、2。

```c
int x[3] = {0, 1, 2};
```

如果给出了初始化式，就可以通过右边元素个数知道数组长度，因此可以省略数组长度。例如上面的表达式和下面的表达式含义相同。

```c
int x[] = {0, 1, 2};
```

如果显式给出了数组长度，而初始化式只给出一部分，剩余元素必须初始化为 0。因此下面两个表达式含义相同。

```c
int x[5] = {1, 2, 3, 0, 0};
int x[5] = {1, 2, 3};
```

另外，作为只有 `char` 数组初始化式才有的特殊语法，允许使用字符串 literal 作为初始化式，如下。

```c
char msg[] = "foo";
```

上面的表达式与下面的表达式含义相同。

```c
char msg[4] = {'f', 'o', 'o', '\0'};
```

## 全局变量的初始化式

全局变量的初始化式需要在编译时计算。计算结果要么是普通字节序列，要么是函数或全局变量的指针。如果是指针，可以带有一个表示该指针 offset 的整数。

完全没有给出初始化式的全局变量，需要初始化为所有 bit 都为 0。这是 C 语法规定的。

如果初始化式无法得到上述计算结果，请作为编译错误处理。

## 局部变量的初始化式

局部变量的初始化式看起来和全局变量的初始化式相同，但意义差别很大。局部变量初始化式是在现场执行的表达式。因此，它的内容不需要在编译时确定。

基本上，像 `int x = 5;` 这样的语句，会被编译成类似分成 `int x; x = 5;` 两个语句书写的形式。

像 `int x[] = {1, 2, foo()};` 这样的语句，会被编译成类似下面的语句。

```c
int x[3];
x[0] = 1;
x[1] = 2;
x[2] = foo();
```

完全没有给出初始化式的局部变量，其内容是不定的。因此不需要初始化这种变量。

### 专栏：word 的大小

在 x86-64 中，“word”这个术语既可以表示 16 bit 数据，也可以表示 64 bit 数据。这种状态有些复杂，但它有历史原因。

“word”这个术语本来指的是计算机上能够自然处理的最大整数或地址的大小。在 x86-64 这个 64 bit 处理器上，把 64 bit 称为 1 word，就是来自这个含义。

另一方面，把 16 bit 称为 1 word，来自 16 bit 处理器 8086 的术语。Intel 工程师把 8086 扩展成 32 bit 并制造 386 处理器时，为了避免“word”的大小发生变化，把 32 bit 称为 double word 或 dword。同样，在把 386 扩展为 64 bit 的 x86-64 中，又把 64 bit 称为 quad word 或 qword。正是这种兼容性考虑，产生了 word 的两个不同含义。
