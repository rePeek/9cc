# 分割编译与链接

到这个阶段为止，我们一直以“一个 C 文件和一个测试用 shell script”的文件结构推进开发。这个结构本身并没有问题，但源代码逐渐变长了，所以在这里把它拆分成多个 C 文件，让整体更清晰。在这个步骤中，会把 `9cc.c` 这一个文件拆成以下 5 个文件。

- main.c: main関数

- `9cc.h`：头文件
- `main.c`：`main` 函数
- `parse.c`：parser
- `codegen.c`：代码生成器

`main` 函数很小，所以也可以放进其他 C 文件中；但从语义上说，它既不属于 `parse.c`，也不属于 `codegen.c`，因此把它分到单独文件中。

本章会先说明分割编译的概念及其意义，然后说明具体步骤。

## 什么是分割编译

### 分割编译及其必要性

分割编译是指把一个程序拆分成多个源文件来写，并分别编译。在分割编译中，编译器读取的不是整个程序，而是程序片段，并输出与之对应的片段。包含程序片段、单独无法执行的文件称为“目标文件”（object file，扩展名为 `.o`）。在分割编译中，最后会把目标文件连接起来，生成一个文件。把目标文件汇总成一个可执行文件的程序称为“linker”。

先理解为什么需要分割编译。实际上，从技术上说，并不存在“必须拆分源代码”的必然性。如果把全部源代码一次性交给编译器，逻辑上编译器可以不借助 linker，直接输出完整的可执行文件。

不过，如果采用这种做法，编译器必须真正知道程序使用的所有代码。例如 `printf` 等标准库函数，通常是标准库作者用 C 写的函数。为了省略链接步骤，这类函数的源代码也必须每次都作为编译器输入提供。反复编译同一个函数，大多数情况下只是浪费时间。因此，标准库通常以已经编译好的目标文件形式发布，本地不需要每次重新编译。也就是说，即使是由一个源文件构成的程序，只要使用标准库，实际上也已经在利用分割编译。

如果不进行分割编译，即使只修改一行，也要重新编译全部代码。几万行长度的代码编译就会花费数十秒。大型项目的源代码可能超过 1000 万行，如果把它作为一个单位编译，一天之内恐怕都无法结束。内存也会需要 100GiB 这样的规模。这种构建流程并不现实。

另外，单纯地把所有函数和变量都写在一个文件中，对人类来说也很难管理。

分割编译正是因为上述原因而成为必要。

### 专栏：linker 的历史

把多个片段式机器码 routine 连接起来，汇总成一个程序，这种 linker 功能从计算机黎明期就已经被需要。1947 年，John Mauchly（第一台数字计算机 ENIAC 的项目负责人）描述过一种程序，它会 relocation 从磁带读取的子程序，并把它们汇总成一个程序。

即使在最早期的计算机中，通用 subroutine 也希望只写一次，然后从各种程序中使用。这样一来，就需要一种 linker，把程序片段结合成可执行程序。1947 年还是汇编器尚未被使用、程序员直接用机器码写代码的时代。因此对程序员来说，linker 实际上是比汇编器更早想要制作的程序。

### 头文件的必要性及其内容

在分割编译中，编译器只会看到程序的一部分代码；但这并不意味着编译器可以编译程序的任意小片段。请考虑下面的代码。

```c
void print_bar(struct Foo *obj) {
  printf("%d\n", obj->bar);
}
```

在上面的代码中，如果知道结构体 `Foo` 的类型，就可以输出与这段代码对应的汇编；否则就无法编译这个函数。

进行分割编译时，需要在各个文件中放入足够的信息，使每个 C 文件都能被编译。话虽如此，如果把其他文件中写着的代码全部写进去，就根本不是分割编译了，所以需要在一定程度上取舍信息。

作为一个例子，来思考为了输出调用另一个 C 文件中函数的代码，需要放入什么信息。编译器需要以下信息。

- 首先，需要知道某个 identifier 是函数名。
- 编译器输出的函数调用代码，会按某个规定顺序把参数设置到寄存器中，然后使用 `call` 指令跳转到另一个函数的开头。根据参数类型，有时还会进行把整数转换成浮点数等操作。如果参数类型或个数错误，也需要显示错误消息。因此需要函数参数的个数以及各个参数的类型。
- 被调用函数内部做了什么，对调用方来说最终只是返回回来，所以编译调用方函数时，不需要被调用函数的代码。
- `call` 跳转目标地址在分割编译时还不知道，但汇编器可以先输出一个暂时跳转到地址 0 的 `call` 指令，并在目标文件中留下“把目标文件第 X 字节处修正为名为 Y 的函数地址”这样的信息。linker 查看这些信息，决定可执行文件的布局后，对程序片段进行 binary patching，修正跳转目标地址（这个操作称为 relocation）。因此，进行分割编译时需要函数名，但不需要函数地址。

总结上述需求可知，只要有省略函数本体 `{ ... }` 的东西，就有足够信息调用该函数。这种省略函数本体的东西称为函数的“声明”（declaration）。声明只是告诉编译器类型和名称，不包含函数代码。例如，下面是 `strncmp` 的声明。

```c
int strncmp(const char *s1, const char *s2, size_t n);
```

编译器看到上面这一行，就能知道 `strncmp` 的存在和它的类型。与声明相对，包含函数代码的东西称为“定义”（definition）。

函数声明也可以加上表示声明的关键字 `extern`：

```c
extern int strncmp(const char *s1, const char *s2, size_t n);
```

这样写也可以。但对于函数来说，由于函数本体被省略就能区分声明和定义，所以也可以不加 `extern`。

另外，参数只要知道类型即可，所以声明中可以省略参数名。不过为了让人更容易理解，通常声明中也会写上参数名。

再以结构体类型为例。如果有两个以上 C 文件使用同一个结构体，就需要在每个 C 文件中写入同一个结构体声明。如果某个结构体只在一个 C 文件中使用，那么其他 C 文件并不需要知道它的存在。

在 C 中，会把这种编译其他 C 文件时需要的声明汇总起来，写进称为头文件（扩展名为 `.h`）的文件中。把声明写在 `foo.h` 中，并在需要它的另一个 C 文件里写上 `#include "foo.h"`，那么 `#include` 那一行就会被替换为 `foo.h` 文件的内容。

`typedef` 等也用于告诉编译器类型信息。如果这类东西被多个 C 文件使用，也需要写进头文件。

编译器读取声明时，并不会特别输出任何汇编。声明是使用其他文件中包含的函数或变量所需的信息，它本身并不定义函数或变量。

理解到这里的分割编译后，应该就能明白“使用 `printf` 时，把 `#include <stdio.h>` 当成咒语写上去”实际上是在做什么。C 标准库会被隐式传给 linker，所以 linker 可以链接包含 `printf` 函数调用的目标文件，并创建可执行文件。另一方面，编译器默认并没有关于 `printf` 的特别知识。`printf` 不是内建函数，也不存在“标准库头文件会自动读取”这样的规格。因此刚启动时，编译器对 `printf` 一无所知。在这种状态下，通过 include C 标准库附带的头文件，编译器才能知道 `printf` 的存在及其类型，并能够编译 `printf` 函数调用。

### 专栏：one-pass compiler 与前向声明

在 C 中，即使把所有函数都写在一个文件里，有时也需要声明。C 语言规格允许编译器不读取整个文件，而是从开头按顺序逐个编译函数。因此，每个函数都必须只用该函数在文件中出现位置之前已经写出的信息就能被编译。因此，如果想使用在文件后方定义的函数，就需要事先写下该函数的声明。这种声明称为“前向声明”（forward declaration）。

通过调整函数写入文件的顺序，大多数前向声明都可以省掉；但如果想写相互递归的函数，前向声明就是必需的。

C 语言这种允许“不读取整个文件就编译”的规格，在主内存非常小的时代有意义；但放到今天，恐怕只能说是过时规格。如果编译器再聪明一些，对于同一个文件中写着的定义，本应可以不写声明。不过，这个行为已经是语言规格的一部分，所以需要记住。

### 链接错误

最后把目标文件汇总交给 linker 时，整体上必须不多不少地包含足以构成程序的信息。

如果程序中只包含函数 `foo` 的声明而没有定义，各个 C 文件即使包含调用 `foo` 的代码，也可以正常编译。但是，当最后 linker 试图创建完整程序时，本应修正为 `foo` 地址的位置，因为没有 `foo` 而无法修正，于是就会出错。

链接时的错误称为链接错误。

如果多个目标文件中包含相同函数或变量，也会发生链接错误。因为对 linker 来说，如果存在重复，它不清楚该选择哪一个。这种重复错误常发生在误把定义写进头文件时。头文件会被多个 C 文件 include，所以如果头文件里有定义，就等同于在多个 C 文件中重复写入定义。为了解决这种错误，请只在头文件中写声明，并把实体移动到某一个 C 文件中。

### 专栏：重复定义与链接错误

存在重复定义时，也可能有一种 linker 行为：选择其中一个并忽略其余定义。对于这种 linker，即使重复定义也不会出错。

在实际目标文件中，也可以按定义选择是否允许重复。inline 函数、C++ template 的展开结果等，会以允许重复的形式包含在目标文件中。目标文件格式和 linker 行为意外地复杂，例外很多；但这类行为终究是例外。默认情况下，重复定义通常会成为错误。

### 全局变量的声明与定义

我们的编译器中还没有全局变量，所以还没有出现对应全局变量的汇编例子。但在汇编层面，全局变量几乎和函数一样。因此，和函数一样，全局变量也有定义和声明的区别。如果变量实体重复存在于多个 C 文件中，通常会成为链接错误。

全局变量默认会被分配到禁止执行的内存区域，所以跳转到那里会导致程序因 segmentation fault 崩溃。但从本质上说，除此之外，数据和代码并不存在差异。运行时也可以把函数当作数据像全局变量那样读取；如果修改内存属性允许执行，并跳转到数据，也可以把数据作为代码执行。

用实际代码确认一下：函数和全局变量本质上都只是存在于内存上的数据。下面的代码中，identifier `main` 被定义为全局变量。`main` 的内容是 x86-64 机器码。

```c
char main[] = "\x48\xc7\xc0\x2a\x00\x00\x00\xc3";
```

把上述 C 代码保存为名为 `foo.c` 的文件并编译，然后用 `objdump` 确认内容。`objdump` 默认只会把全局变量内容以十六进制显示；但传入 `-D` 选项后，可以强行把数据当作代码反汇编。

```console
$ cc -c foo.c
$ objdump -D -M intel foo.o
Disassembly of section .data:

0000000000000000 <main>:
   0:   48 c7 c0 2a 00 00 00    mov      rax,0x2a
   7:   c3                      ret
```

“数据会被映射到禁止执行区域”这一默认行为，可以通过编译时传入 `-Wl,--omagic` 选项来改变。用这个选项生成可执行文件试试看。

```console
$ cc -static -Wl,--omagic -o foo foo.o
```

函数和变量在汇编中都只是标签，并属于同一个命名空间，所以 linker 汇总多个目标文件时，并不关心哪个是函数、哪个是数据。因此，即使 `main` 在 C 层面被定义为数据，linker 也会像 `main` 是函数一样成功链接。

运行生成的文件试试看。

```console
$ ./foo
$ echo $?
42
```

如上所示，正确返回了 42。也就是说，名为 `main` 的全局变量内容被作为代码执行了。

在 C 语法中，对于全局变量，加上 `extern` 就会变成声明。下面是 `int` 类型全局变量 `foo` 的声明。

```c
extern int foo;
```

编写包含 `foo` 的程序时，会把上面这一行写进头文件。然后在某一个 C 文件中定义 `foo`。下面是 `foo` 的定义。

```c
int foo;
```

另外，在 C 中，没有给出 initializer 的全局变量会被初始化为 0。因此，这种变量在语义上等同于用 `0`、`{0, 0, ...}`、`"\0\0\0\0..."` 等初始化。

如果像 `int foo = 3` 这样写 initializer，请只在定义中写 initializer。声明只是为了告诉编译器变量类型，不需要具体 initializer。编译器看到全局变量声明时并不会特别输出汇编，所以当场并不需要知道其内容如何初始化。

当 initializer 被省略时，全局变量的声明和定义只有是否存在 `extern` 的差异，外观看起来很像；但声明和定义是不同的东西。请在这里明确掌握这一点。

### 专栏：Intel CPU 的 F00F bug

1997 年以前的 Intel Pentium 处理器中，有一个严重 bug：执行 `F0 0F C7 C8` 这 4 字节指令时，CPU 会完全 hang 住。

正式来说，并不存在与这 4 字节指令对应的汇编指令；但如果强行写成汇编，会是 `lock cmpxchg8b eax` 这条指令。`0F C7 C8` 是 `cmpxchg8b eax` 指令，它会在寄存器与内存之间 atomic 地交换 8 字节值（即使在多核中，其他核也无法观察到中间状态）。`F0` 是称为 `lock` prefix 的附加信息，具有让后续指令 atomic 的效果。但是，`cmpxchg8b` 原本就是 atomic 的，所以 `lock cmpxchg8b eax` 是冗余且非法的指令写法。因此，语法上不存在这样的汇编指令，`F0 0F C7 C8` 这一字节序列不会出现在普通程序中，Intel 也就没能在处理器大量生产前发现这个 bug。

使用“把 `main` 函数写成数据”这个 hack，复现 F00F bug 的代码可以用 C 写成下面一行。

```c
char main[] = "\xf0\x0f\xc7\xc8";
```

在现代 x86 上，这个函数是无害的；但在 1997 年当时的 Pentium 上，任何人都能用这一行程序轻易让整个系统 hang 住。

如果是个人完全独占的 PC，F00F bug 并不是大问题；但如果是像今天云计算这样共享 CPU 的使用方式，这个 bug 就是致命的。起初，人们认为 F00F bug 无法修复，只能回收更换 CPU；但之后产生了一种在 OS kernel 的 exception handler 层面用 tricky 方法规避 bug 的手段，对 Intel 来说幸运的是，最终避免了产品更换。

## 步骤 8：文件拆分与 Makefile 修改

### 文件拆分

请按照本章开头展示的结构拆分文件。`9cc.h` 是头文件。根据程序结构，有时会为每个 `.c` 文件准备一个 `.h` 文件；但多余声明并没有特别危害，所以这里不需要做那么细的依赖关系管理。请准备一个名为 `9cc.h` 的文件，并在所有 C 文件中用 `#include "9cc.h"` include 它。

### 修改 Makefile

既然程序已经改成多个文件，也更新一下 `Makefile`。下面的 `Makefile` 用于编译并链接当前目录中的所有 `.c` 文件，创建名为 `9cc` 的可执行文件。这里假设项目头文件只有一个名为 `9cc.h` 的文件，并且所有 `.c` 文件都会 include 这个头文件。

```makefile
CFLAGS=-std=c11 -g -static
SRCS=$(wildcard *.c)
OBJS=$(SRCS:.c=.o)

9cc: $(OBJS)
        $(CC) -o 9cc $(OBJS) $(LDFLAGS)

$(OBJS): 9cc.h

test: 9cc
        ./test.sh

clean:
         rm -f 9cc *.o *~ tmp*

.PHONY: test clean
```

请注意，`Makefile` 的缩进必须是 tab 字符。

`make` 是功能丰富的工具，未必需要完全掌握；但如果能读懂上面这种程度的 `Makefile`，在很多场景中都会有用。因此本节说明上面的 `Makefile`。

在 `Makefile` 中，由冒号分隔的行，以及 0 行以上用 tab 缩进的命令行，构成一条规则。冒号前面的名称称为“target”。冒号后面的 0 个以上文件名称为依赖文件。

执行 `make foo` 时，`make` 会尝试创建名为 `foo` 的文件。如果指定 target 的文件已经存在，只有当 target 文件比依赖文件更旧时，`make` 才会重新执行 target 规则。这样就实现了“只有源代码被修改时才重新生成 binary”这类行为。

`.PHONY` 是表示 dummy target 的特殊名称。`make test` 和 `make clean` 并不是为了创建名为 `test` 或 `clean` 的文件而执行的；但普通情况下 `make` 并不知道这一点，所以如果偶然存在名为 `test` 或 `clean` 的文件，`make test` 或 `make clean` 就会什么也不做。对于这种 dummy target，通过在 `.PHONY` 中指定，可以告诉 `make`：并不是真的想创建这个名字的文件，而是不管指定 target 的文件是否存在，都应该执行规则中的命令。

`CFLAGS`、`SRCS`、`OBJS` 是变量。

`CFLAGS` 是会被 `make` 内置规则识别的变量，用来写传给 C 编译器的命令行选项。这里传入以下 flag。

- `-std=c11`：告知源代码是用 C 的较新标准 C11 编写的
- `-g`：输出调试信息
- `-static`：进行静态链接

`SRCS` 右边使用的 `wildcard` 是 `make` 提供的函数，会展开为匹配函数参数的文件名。`$(wildcard *.c)` 目前会展开为 `main.c parse.c codegen.c`。

`OBJS` 的右边使用了变量替换规则，由此生成把 `SRCS` 中 `.c` 替换为 `.o` 后的值。`SRCS` 是 `main.c parse.c codegen.c`，所以 `OBJS` 会变成 `main.o parse.o codegen.o`。

在理解这些之后，追踪一下执行 `make 9cc` 时会发生什么。`make` 会尝试生成作为参数指定的 target，所以创建 `9cc` 文件就是命令的最终目标（如果没有参数，会选择第一条规则，所以这里不指定 `9cc` 也可以）。为此，`make` 会沿着依赖关系追踪，尝试构建缺失或已经变旧的文件。

`9cc` 的依赖文件，是当前目录中 `.c` 文件对应的 `.o` 文件。如果上次执行 `make` 时留下的 `.o` 文件仍在，并且它的 timestamp 比对应的 `.c` 文件更新，`make` 不会特意重新执行同一个命令。只有当 `.o` 文件不存在，或者 `.c` 文件更新时，才会执行编译器生成 `.o` 文件。

`$(OBJS): 9cc.h` 这条规则表示所有 `.o` 文件都依赖 `9cc.h`。因此，如果修改了 `9cc.h`，所有 `.o` 文件都会被重新编译。

### 专栏：`static` 关键字的各种含义

C 的 `static` 关键字主要有以下两种用途。

1. 给局部变量加上 `static`，使其在退出函数后仍保存值
2. 给全局变量或函数加上 `static`，把该变量或函数的 scope 变成文件 scope

这两种用途并没有特别共通性，却使用同一个关键字，因此成为学习 C 时容易混乱的点之一。理想情况下，用途 1 也许应该使用 `persistent`，用途 2 使用 `private` 等不同关键字。更理想地说，对于用途 2，也许应该默认 `private`，并给全局 scope 的变量和函数加上 `public`。

C 重复使用关键字的原因，是为了兼容过去写下的代码资产。如果向语言中添加 `private` 等新关键字，那么把这些关键字用作变量或函数名的既有程序就无法编译。C 不希望发生这种事，所以没有增加关键字，而是选择在不同上下文中复用既有关键字。

如果在 1970 年代某个阶段，做出“增加新关键字而不是复用 `static` 关键字”的决定，应该不需要修改太多代码；但如果思考“换成自己会怎么做”，这也是一个相当困难的问题。
