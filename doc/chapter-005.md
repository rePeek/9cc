# 函数与局部变量

本章会实现函数和局部变量。同时，也会实现简单的控制结构。本章结束后，就能够编译下面这样的代码。

```c
// 计算从 m 到 n 的和
sum(m, n) {
  acc = 0;
  for (i = m; i <= n; i = i + 1)
    acc = acc + i;
  return acc;
}

main() {
  return sum(1, 10); // 返回 55
}
```

上面的代码与 C 还存在差距，但即便如此，也已经可以说相当接近 C 了。

## 步骤 9：单字符局部变量

到上一章为止，我们已经能制作支持四则运算语言的编译器。本节会给这门语言添加功能，让它能够使用变量。具体目标是能够编译下面这样包含变量的多条语句。

```c
a = 3;
b = 5 * 6 - 8;
a + b / 2;
```

这里约定最后一个表达式的结果作为整个程序的计算结果。与只有四则运算的语言相比，这门语言是不是已经明显有了“真正语言”的感觉？

本章先说明应该如何实现变量，然后再增量式地实现变量。

### 栈上的变量区域

C 中的变量存在于内存上。也可以说，变量是给内存地址取的名字。通过给内存地址命名，就可以不再说“访问内存的 `0x6080` 地址”，而是说“访问变量 `a`”。

不过，函数的局部变量必须在每次函数调用中分别存在。只考虑实现方便的话，似乎可以把地址写死，例如“函数 `f` 的局部变量 `a` 放在 `0x6080` 地址”。但这样在递归调用 `f` 时就无法正常工作。为了让局部变量在每次函数调用中分别拥有一份，C 会把局部变量放在栈上。

用一个具体例子思考栈的内容。假设有一个拥有局部变量 `a` 和 `b` 的函数 `f`，另一个某函数调用了 `f`。函数调用的 `call` 指令会把返回地址压入栈中，因此 `f` 被调用时的栈顶存放着这个返回地址。除此之外，栈中原本还存在一些值。这里具体值不重要，用 `......` 表示。画成图如下。

```text
......
返回地址   <- RSP
```

这里用 `<- RSP` 表示当前 RSP 寄存器的值指向该地址。假设 `a` 和 `b` 的大小分别是 8 字节。

栈向下增长。从这个状态开始，为了确保 `a` 和 `b` 的区域，需要把 RSP 向下推两个变量的大小，也就是合计 16 字节。执行后如下。

```text
......
返回地址
a
b                 <- RSP
```

采用上述布局后，使用 `RSP+8` 的值可以访问 `a`，使用 `RSP` 的值可以访问 `b`。这种每次函数调用时分配的内存区域称为“函数帧”（function frame）或“activation record”。

RSP 要修改多少字节、在这样确保的区域中以什么顺序放置变量，这些对其他函数不可见，所以可以根据编译器实现方便任意决定。

局部变量基本上就是以这种简单方式实现的。

不过，这种方法有一个缺点，所以实际实现中还会使用另一个寄存器。请回想一下，在我们的编译器中（以及其他编译器中），函数执行期间 RSP 可能会被修改。`9cc` 会把表达式的中间计算结果 push/pop 到使用 RSP 的栈上，因此 RSP 的值会频繁变化。所以，不能用从 RSP 出发的固定 offset 来访问 `a` 或 `b`。

解决这个问题的常见做法，是准备一个不同于 RSP 的寄存器，让它始终指向当前函数帧的开始位置。这种寄存器称为“基址寄存器”，其中保存的值称为“基址指针”。在 x86-64 中，惯例上使用 RBP 寄存器作为基址寄存器。

函数执行期间，基址指针不能变化（这正是准备基址指针的理由）。如果从某函数调用另一个函数，返回时基址指针变成了别的值，那就不行了。因此，每次函数调用时都需要保存原来的基址指针，并在返回前写回。

下面的图展示使用基址指针的函数调用中栈的状态。假设拥有局部变量 `x` 和 `y` 的函数 `g` 调用 `f`。在 `g` 执行期间，栈如下。

```text
......
g 的返回地址
g 的调用时的 RBP   <- RBP
x
y                      <- RSP
```

从这里调用 `f` 后，状态如下。

```text
......
g 的返回地址
g 的调用时的 RBP
x
y
f 的返回地址
f 的调用时的 RBP   <- RBP
a
b                      <- RSP
```

这样一来，就总是可以通过 `RBP-8` 这个地址访问 `a`，通过 `RBP-16` 这个地址访问 `b`。具体思考创建这种栈状态的汇编，编译器只需要在每个函数开头输出下面这样的汇编。

```asm
push rbp
mov rbp, rsp
sub rsp, 16
```

这种由编译器在函数开头输出的固定形式指令称为“prologue”。另外，这里的 16 实际上需要根据每个函数中变量的个数和大小来决定。

从 RSP 指向返回地址的状态开始执行上面的代码，就会创建出符合预期的函数帧。下面按每条指令展示栈状态。

1. 刚通过 `call` 调用 `f` 后的栈

```text
......
g 的返回地址
g 的调用时的 RBP   <- RBP
x
y
f 的返回地址     <- RSP
```

2. 执行 `push rbp` 后的栈

```text
......
g 的返回地址
g 的调用时的 RBP   <- RBP
x
y
f 的返回地址
f 的调用时的 RBP   <- RSP
```

3. 执行 `mov rbp, rsp` 后的栈

```text

g 的返回地址
g 的调用时的 RBP
x

f 的返回地址
f 的调用时的 RBP   <- RSP, RBP
```

4. 执行 `sub rsp, 16` 后的栈

```text

g 的调用时的 RBP
x
y
f 的返回地址
f 的调用时的 RBP   <- RBP
a
b                      <- RSP
```

从函数返回时，需要把 RBP 恢复成原来的值，让 RSP 指向返回地址，然后调用 `ret` 指令（`ret` 指令会从栈中 pop 地址并跳转到那里）。执行这些操作的代码可以简洁地写成下面这样。

```asm
mov rsp, rbp
pop rbp
ret
```

这种由编译器在函数末尾输出的固定形式指令称为“epilogue”。

下面展示执行 epilogue 时的栈状态。RSP 所指地址之下的栈区域已经可以视为无效数据，所以图中省略。

1. 执行 `mov rsp, rbp` 前的栈

```text

g 的返回地址
g 的调用时的 RBP
x
y
f 的返回地址
f 的调用时的 RBP   <- RBP
a
b                      <- RSP
```

2. 执行 `mov rsp, rbp` 后的栈

```text
......

g 的调用时的 RBP
x
y
f 的返回地址
f 的调用时的 RBP   <- RSP, RBP
```

3. 执行 `pop rbp` 后的栈

```text
......
g 的返回地址
g 的调用时的 RBP   <- RBP
x
y
f 的返回地址     <- RSP
```

4. 执行 `ret` 后的栈

```text
......
g 的返回地址
g 的调用时的 RBP   <- RBP
x
y                      <- RSP
```

像这样，执行 epilogue 会恢复调用方函数 `g` 的栈状态。`call` 指令会把 `call` 指令自身的下一条指令地址压入栈。epilogue 中的 `ret` 会 pop 该地址并跳转到那里，所以函数 `g` 会从 `call` 后面的下一条指令恢复执行。这种行为与我们熟悉的函数行为完全一致。

函数调用和函数局部变量就是这样实现的。

### 专栏：栈增长的方向

如上所述，x86-64 的栈从高地址向低地址增长。反方向，也就是栈向上增长，感觉似乎更自然；那么为什么栈被设计为向下增长呢？

实际上，栈向下增长并没有技术上的必然性。实际 CPU 和 ABI 中，主流做法是把栈起点放在高地址并向下增长；但虽然极其小众，也存在栈向反方向增长的架构。例如 8051 microcontroller、PA-RISC ABI、Multics 等中，栈会向高地址方向增长。

话虽如此，栈向下增长这种设计也并不特别不自然。

刚上电后，CPU 从空白状态开始执行程序时，执行开始地址通常由 CPU 规格决定。常见设计中，CPU 会从地址 0 这样的低地址开始执行。这样一来，程序代码通常会集中放在低地址。为了避免栈增长后与程序代码重叠，如果把两者尽量分开放置，就会把栈放在高地址，并设计为朝地址空间中央方向增长。这样一来，栈就会向下增长。

当然，也可以考虑与上述 CPU 不同的设计，那样栈向上增长会是更自然的布局。坦白说，这个问题哪种都可以。实际情况只是：行业的一般认知中，机器栈是向下增长的。

### tokenizer 的修改

既然已经知道变量该如何实现，就立刻动手实现吧。不过，支持任意数量的变量会一下子变得太难，所以在这个步骤中，我们把变量限制为一个小写字母。变量 `a` 固定在 `RBP-8`，变量 `b` 固定在 `RBP-16`，变量 `c` 固定在 `RBP-24`，以此类推，假定所有变量总是存在。英文字母有 26 个，所以在调用函数时让 `RSP` 下移 `26 * 8`，也就是 208 字节，就能为所有单字符变量分配区域。

开始实现。首先修改 tokenizer，让它除了目前已有的语法元素之外，还能把单字符变量切分成 token。为此需要增加一种新的 token 类型。变量名可以从 `str` 成员读取，因此不需要给 `Token` 类型额外增加成员。结果，token 类型会变成下面这样。

```c
enum {
  TK_RESERVED, // 符号
  TK_IDENT,    // 識別子
  TK_NUM,      // 整数 token
  TK_EOF,      // 表示输入结束的 token
} TokenKind;
```

修改 tokenizer，使它在遇到小写英文字母时创建 `TK_IDENT` 类型的 token。给 tokenizer 增加下面这样的 `if` 语句即可。

```c
if ('a' <= *p && *p <= 'z') {
  cur = new_token(TK_IDENT, cur, p++);
  cur->len = 1;
  continue;
}
```

### parser 的修改

在递归下降语法分析中，只要知道语法，就可以机械地映射到函数调用。因此，要思考 parser 需要加入哪些修改，就必须先考虑加入变量名，也就是标识符之后的新语法是什么样的。

把标识符记作 `ident`。它和 `num` 一样是终结符。变量可以用在数值能出现的任何地方，所以把原先的 `num` 改成 `num | ident`，就能得到变量和数值可以出现在同样位置的语法。

此外，还需要给语法加入赋值表达式。变量如果不能赋值就没有意义，所以我们希望语法允许 `a=1` 这样的表达式。这里为了和 C 保持一致，也允许写成 `a=b=1` 这样的形式。

另外，还希望能用分号分隔写多个语句（statement），因此新的语法如下。

```ebnf
program    = stmt*
stmt       = expr ";"
expr       = assign
assign     = equality ("=" assign)?
equality   = relational ("==" relational | "!=" relational)*
relational = add ("<" add | "<=" add | ">" add | ">=" add)*
add        = mul ("+" mul | "-" mul)*
mul        = unary ("*" unary | "/" unary)*
unary      = ("+" | "-")? primary
primary    = num | ident | "(" expr ")"
```

先确认 `42;` 或 `a=b=2; a+b;` 这样的程序符合这个语法。然后修改到目前为止写好的 parser，让它能够解析上面的语法。在这个阶段，`a+1=5` 这样的表达式也会被成功解析，但这是正确的行为。排除这种语义上非法的表达式，是下一个处理阶段要做的事情。改造 parser 没有特别复杂的地方，像之前一样把语法元素直接映射为函数调用即可。

因为现在可以用分号分隔写多个表达式，所以解析结果中会有多个节点，需要把它们保存在某个地方。现在先准备下面这个全局数组，把解析结果节点依次存进去。最后一个节点填入 `NULL`，就能知道哪里是末尾。下面展示一部分新加入的代码。

```c
Node *code[100];

Node *assign() {
  Node *node = equality();
  if (consume("="))
    node = new_node(ND_ASSIGN, node, assign());
  return node;
}

Node *expr() {
  return assign();
}

Node *stmt() {
  Node *node = expr();
  expect(";");
  return node;
}

void program() {
  int i = 0;
  while (!at_eof())
    code[i++] = stmt();
  code[i] = NULL;
}
```

抽象语法树也需要能够表示一种新的节点：“表示局部变量的节点”。为此，给局部变量增加一种新的类型，并给节点增加新的成员。例如可以写成下面这样。使用这个数据结构时，parser 会针对标识符 token 创建并返回 `ND_LVAR` 类型的节点。

```c
typedef enum {
  ND_ADD,    // +
  ND_SUB,    // -
  ND_MUL,    // *
  ND_DIV,    // /
  ND_ASSIGN, // =
  ND_LVAR,   // 局部变量
  ND_NUM,    // 整数
} NodeKind;

typedef struct Node Node;

// 抽象语法树节点
struct Node {
  NodeKind kind; // 节点类型
  Node *lhs;     // 左辺
  Node *rhs;     // 右辺
  int val;       // 仅在 kind 为 ND_NUM 时使用
  int offset;    // 仅在 kind 为 ND_LVAR 时使用
};
```

`offset` 成员表示局部变量相对于基址指针的偏移量。目前变量 `a` 位于 `RBP-8`，`b` 位于 `RBP-16`，以此类推，局部变量的位置由名字固定决定，因此 offset 可以在语法分析阶段确定。下面的代码读取标识符并返回 `ND_LVAR` 类型的节点。

```c
Node *primary() {
  ...

  Token *tok = consume_ident();
  if (tok) {
    Node *node = calloc(1, sizeof(Node));
    node->kind = ND_LVAR;
    node->offset = (tok->str[0] - 'a' + 1) * 8;
    return node;
  }

  ...
}
```

#### 专栏：ASCII 码

在 ASCII 码中，0 到 127 的数字被分配给字符。下面是 ASCII 码中的字符分配表。

```text
  0  NUL SOH STX ETX EOT ENQ ACK BEL
  8   BS  HT  NL  VT  NP  CR  SO  SI
 16  DLE DC1 DC2 DC3 DC4 NAK SYN ETB
 24  CAN  EM SUB ESC  FS  GS  RS  US
 32   sp   !   "   #   $   %   &   '
 40    (   )   *   +   ,   -   .   /
 48    0   1   2   3   4   5   6   7
 56    8   9   :   ;   <   =   >   ?
 64    @   A   B   C   D   E   F   G
 72    H   I   J   K   L   M   N   O
 80    P   Q   R   S   T   U   V   W
 88    X   Y   Z   [   \   ]   ^   _
 96    `   a   b   c   d   e   f   g
104    h   i   j   k   l   m   n   o
112    p   q   r   s   t   u   v   w
120    x   y   z   {   |   }   ~ DEL
```

0 到 31 是控制字符。现在除了 `NUL` 字符和换行字符等之外，几乎没有机会使用这些控制字符，大多数控制字符只是占据了字符编码中的重要位置却很少发挥作用。但是在 1963 年制定 ASCII 码时，这些控制字符确实经常被使用。在制定 ASCII 标准时，甚至有人提议不要加入小写英文字母，而是加入更多控制字符。

数字被分配到 48 到 58，大写字母被分配到 65 到 90，小写字母被分配到 97 到 122。注意，这些字符被分配到了连续的编码。也就是说，`0123456789` 和 `abcdefg...` 在字符编码上是连续的。把有顺序的字符放在连续位置上看起来理所当然，但在当时主流的 EBCDIC 等字符编码中，受打孔卡片影响，英文字母在编码上并不是连续的。

在 C 中，字符只是整数类型中的一个小值，写成与该字符对应的编码数字在意义上没有区别。假设使用 ASCII，例如 `'a'` 等价于 97，`'0'` 等价于 48。上面的代码里有一个从字符中以数值形式减去 `a` 的表达式。这样做可以计算给定字符距离 `a` 有几个字符。这个技巧正是因为英文字母在 ASCII 码中连续排列才成立。

### 左值与右值

赋值表达式和其他二元运算符不同，需要特别处理左边的值，因此这里先说明这一点。

赋值表达式的左边并不是任何表达式都可以。例如，不能写 `1=2` 来把 `1` 变成 `2`。像 `a=2` 这样的赋值是允许的，但 `(a+1)=2` 这样的语句是非法的。9cc 目前还没有指针和结构体，但如果有的话，像 `*p=2` 这样对指针所指位置赋值，或者像 `a.b=2` 这样对结构体成员赋值，都必须作为合法表达式允许。那么，应该如何区分合法表达式和非法表达式呢？

这个问题有一个简单答案。在 C 中，能够出现在赋值表达式左边的，基本上只有能够指定内存地址的表达式。

变量存在于内存中并拥有地址，所以变量可以写在赋值左边。同样，像 `*p` 这样的指针引用，说的是把 `p` 的值作为地址，因此也可以写在左边。像 `a.b` 这样的结构体成员访问，指向的是从内存中结构体 `a` 的起始位置前进成员 `b` 的偏移量后得到的内存地址，因此也可以写在左边。

另一方面，像 `a+1` 这样的表达式结果不是变量，所以不能作为指定内存地址的表达式来使用。这类临时值实际上可能只存在于寄存器中，并不在内存上；即使存在于内存中，通常也不能通过相对某个已知变量的固定偏移来访问。基于这些理由，即使写成 `&(a+1)`，也不允许取得 `a+1` 结果的地址，而会变成编译错误。这样的表达式不能写在赋值语句的左边。

可以写在左边的值称为左值（left value），也叫 `lvalue`；不能写在左边的值称为右值（right value），也叫 `rvalue`。在我们当前的语言中，只有变量是左值，其他所有值都是右值。

在为变量生成代码时，可以从左值出发思考。如果变量作为赋值左边出现，就把左边的值生成为变量地址，然后把右边的求值结果 store 到这个地址。这样就能实现赋值表达式。如果变量出现在其他上下文中，也先同样计算变量地址，然后从该地址 load 出值，把左值转换成右值。这样就能取得变量的值。

### 从任意地址加载值的方法

到目前为止，代码生成只访问了栈顶的内存，但局部变量需要访问栈上的任意位置。这里说明访问内存的方法。

CPU 不仅可以访问栈顶，也可以从内存中的任意地址 load 值，或者向任意地址 store 值。

从内存 load 值时，使用 `mov dst, [src]` 这个语法。该指令的意思是：“把 `src` 寄存器中的值视为地址，从那里 load 一个值并保存到 `dst`”。例如，`mov rdi, [rax]` 表示从 `RAX` 中保存的地址 load 值，并设置到 `RDI`。

store 时，使用 `mov [dst], src` 这个语法。该指令的意思是：“把 `dst` 寄存器中的值视为地址，把 `src` 寄存器的值 store 到那里”。例如，`mov [rdi], rax` 表示把 `RAX` 的值 store 到 `RDI` 中保存的地址。

由于 `push` 和 `pop` 会隐式地把 `RSP` 视为地址来访问内存，所以其实可以用普通的内存访问指令把它们改写成多条指令。也就是说，例如 `pop rax` 等价于下面两条指令。

```asm
mov rax, [rsp]
add rsp, 8
```

而 `push rax` 等价于下面两条指令。

```asm
sub rsp, 8
mov [rsp], rax
```

### code generator 的修改

使用到目前为止的知识，修改 code generator，使它能够处理包含变量的表达式。这次修改会增加一个“把表达式作为左值求值”的函数。下面代码中的 `gen_lval` 函数就是做这件事的。当给定节点指向变量时，`gen_lval` 会计算该变量的地址，并把它 push 到栈上。除此以外的情况则显示错误。这样就能排除 `(a+1)=2` 这样的表达式。

把变量作为右值使用时，先把它作为左值求值，然后把栈顶的计算结果视为地址，并从该地址 load 值。代码如下。

```c
void gen_lval(Node *node) {
  if (node->kind != ND_LVAR)
    error("赋值左边不是变量");

  printf("    mov rax, rbp\n");
  printf("    sub rax, %d\n", node->offset);
  printf("    push rax\n");
}

void gen(Node *node) {
  switch (node->kind) {
  case ND_NUM:
    printf(" push %d\n", node->val);
    return;
  case ND_LVAR:
    gen_lval(node);
    printf(" pop rax\n");
    printf(" mov rax, [rax]\n");
    printf(" push rax\n");
    return;
  case ND_ASSIGN:
    gen_lval(node->lhs);
    gen(node->rhs);

    printf("    pop rdi\n");
    printf("    pop rax\n");
    printf("    mov [rax], rdi\n");
    printf("    push rdi\n");
    return;
  }

  gen(node->lhs);
  gen(node->rhs);

  printf("      pop rdi\n");
  printf("      pop rax\n");

  switch (node->kind) {
  case '+':
    printf(" add rax, rdi\n");
    break;
  case '-':
    printf(" sub rax, rdi\n");
    break;
  case '*':
    printf("      imul rax, rdi\n");
    break;
  case '/':
    printf("      cqo\n");
    printf("      idiv rdi\n");
  }

  printf("      push rax\n");
}
```

### main 函数的修改

现在所有部件都准备好了，也修改 `main` 函数，实际运行编译器吧。

```c
int main(int argc, char **argv) {
  if (argc != 2) {
    error("参数个数不正确");
    return 1;
  }

  // tokenize 并 parse
  // 结果保存到 code
  user_input = argv[1];
  tokenize();
  program();

  // 输出汇编前半部分
  printf(".intel_syntax noprefix\n");
  printf(".globl main\n");
  printf("main:\n");

  // prologue
  // 确保 26 个变量的区域
  printf(" push rbp\n");
  printf(" mov rbp, rsp\n");
  printf(" sub rsp, 208\n");

  // 从第一个表达式开始依次生成代码
  for (int i = 0; code[i]; i++) {
    gen(code[i]);

    // 表达式求值结果会在栈上留下一个值
    // 因此预先 pop，避免栈溢出
    printf(" pop rax\n");
  }

  // epilogue
  // 最后一个表达式的结果留在 RAX 中，因此它就是返回值
  printf(" mov rsp, rbp\n");
  printf(" pop rbp\n");
  printf(" ret\n");
  return 0;
}
```

## 步骤 10：多字符局部变量

前面的章节中，我们把变量名固定为 1 个字符，并把 `a` 到 `z` 的 26 个局部变量当作总是存在来处理。本节将支持名字长度超过 1 个字符的标识符，使下面这样的代码也能编译。

```c
foo = 1;
bar = 2 + 3;
return foo + bar; // 返回 6
```

这里假定变量可以不经定义直接使用。因此在 parser 中，需要针对每个标识符判断它以前是否出现过；如果是新的标识符，就自动在栈区域中为它分配变量。

首先修改 tokenizer，让它把由多个字符组成的标识符读取为 `TK_IDENT` 类型的 token。

变量用链表表示。用名为 `LVar` 的结构体表示一个变量，并用名为 `locals` 的指针持有链表的第一个元素。用代码表示如下。

```c
typedef struct LVar LVar;

// 局部变量的类型
struct LVar {
  LVar *next; // 下一个变量或 NULL
  char *name; // 变量名
  int len;    // 名称长度
  int offset; // 相对 RBP 的 offset
};

// 局部变量
LVar *locals;
```

在 parser 中，如果出现 `TK_IDENT` 类型的 token，就检查这个标识符以前是否出现过。通过遍历 `locals` 并查看变量名，就可以知道它是否是已有变量。如果变量以前出现过，就直接使用该变量的 `offset`。如果是新变量，就创建新的 `LVar`，设置新的 offset，并使用这个 offset。

下面展示按名字查找变量的函数。

```c
// 按名称搜索变量。找不到时返回 NULL。
LVar *find_lvar(Token *tok) {
  for (LVar *var = locals; var; var = var->next)
    if (var->len == tok->len && !memcmp(tok->str, var->name, var->len))
      return var;
  return NULL;
}
```

在 parser 中加入下面这样的代码即可。

```c
Token *tok = consume_ident();
if (tok) {
  Node *node = calloc(1, sizeof(Node));
  node->kind = ND_LVAR;

  LVar *lvar = find_lvar(tok);
  if (lvar) {
    node->offset = lvar->offset;
  } else {
    lvar = calloc(1, sizeof(LVar));
    lvar->next = locals;
    lvar->name = tok->str;
    lvar->len = tok->len;
    lvar->offset = locals->offset + 8;
    node->offset = lvar->offset;
    locals = lvar;
  }
  return node;
}
```

## 步骤 11：`return` 语句

这一节加入 `return` 语句，使下面这样的代码能够编译。

```c
a = 3;
b = 5 * 6 - 8;
return a + b / 2;
```

这里允许把 `return` 语句写在程序中途。和普通 C 一样，程序执行会在第一个 `return` 处中断，并从函数返回。例如下面的程序会返回第一个 `return` 的值，也就是 5。

```c
return 5;
return 8;
```

为了实现这个功能，先考虑加入 `return` 后的语法会变成什么样。到目前为止，statement 只是普通表达式；而在新语法中，要允许 `return <表达式>;`。因此新语法如下。

```ebnf
program = stmt*
stmt    = expr ";"
        | "return" expr ";"
...
```

为了实现它，需要分别对 tokenizer、parser 和 code generator 做一些修改。

首先让 tokenizer 能够识别 `return` 这个 token，并用 `TK_RETURN` 类型表示它。像 `return`、`while`、`int` 这样在语法上具有特殊意义的 token 称为关键字。关键字的数量有限，所以像这样给每种关键字分配单独的 token 类型会更简单。

判断下一个 token 是否是 `return`，看起来只需要检查 tokenizer 剩余输入字符串是否以 `return` 开头即可。但那样会把 `returnx` 这样的 token 错误地切分为 `return` 和 `x`。因此这里除了确认输入开头是 `return` 之外，还需要确认它后面的字符不是构成 token 的字符。

下面展示一个函数，用来判断给定字符是否是构成 token 的字符，也就是是否为英数字或下划线。

```c
int is_alnum(char c) {
  return ('a' <= c && c <= 'z') ||
         ('A' <= c && c <= 'Z') ||
         ('0' <= c && c <= '9') ||
         (c == '_');
}
```

使用这个函数，在 `tokenize` 中加入下面的代码，就能把 `return` 作为 `TK_RETURN` 进行 tokenize。

```c
if (strncmp(p, "return", 6) == 0 && !is_alnum(p[6])) {
  tokens[i].ty = TK_RETURN;
  tokens[i].str = p;
  i++;
  p += 6;
  continue;
}
```

接下来修改 parser，让它能解析包含 `TK_RETURN` 的 token 序列。为此，先添加表示 `return` 语句的节点类型 `ND_RETURN`。然后修改读取 statement 的函数，使它能够解析 `return` 语句。和之前一样，把语法直接映射为函数调用即可完成语法分析。新的 `stmt` 函数如下。

```c
Node *stmt() {
  Node *node;

  if (consume(TK_RETURN)) {
    node = calloc(1, sizeof(Node));
    node->kind = ND_RETURN;
    node->lhs = expr();
  } else {
    node = expr();
  }

  if (!consume(';'))
    error_at(tokens[pos].str, "不是 ';' token");
  return node;
}
```

`ND_RETURN` 类型的节点只在这里生成，所以这里没有新建函数，而是当场 `malloc` 并设置值。

最后修改 code generator，让它针对 `ND_RETURN` 类型的节点输出合适的汇编代码。下面展示新的 `gen` 函数的一部分。

```c
void gen(Node *node) {
  if (node->kind == ND_RETURN) {
    gen(node->lhs);
    printf(" pop rax\n");
    printf(" mov rsp, rbp\n");
    printf(" pop rbp\n");
    printf(" ret\n");
    return;
  }
  ...
}
```

上面代码中的 `gen(node->lhs)` 函数调用会输出作为 `return` 返回值的表达式代码。该代码应该会在栈顶留下一个值。`gen(node->lhs)` 后面的汇编会把这个值从栈中 pop 出来并设置到 `RAX`，然后从函数返回。

到上一章为止实现的功能中，函数最后一定会输出一条 `ret` 指令。按照本节说明的方法实现 `return` 语句后，每个 `return` 语句还会额外输出一条 `ret` 指令。这些指令也可以合并，但这里为了简单实现，允许输出多条 `ret` 指令。现在纠结这些细节没有太大意义，优先保持实现简单很重要。能写复杂代码是一项有用技能，但一开始就不把代码写得过于复杂，有时是更加有用的技能。

#### 专栏：语法的层级

为了判断输入是否符合某种规则，经常会使用“正则表达式”。但是复杂到一定程度以上的语法无法用正则表达式表示。例如，原则上无法写出一个正则表达式来判断字符串中的括号是否匹配。

上下文无关文法（可以用 BNF 表示的文法）比正则表达式更强，例如可以只表示括号匹配的字符串（用 BNF 写作 `S → SS | "(" S ")" | ε`）。但是，和正则表达式一样，上下文无关文法也有局限，无法表示普通编程语言中出现的复杂规则。例如，“变量必须在使用前声明”是 C 语法的一部分，但这样的规则无法用上下文无关文法表示。

如果写出了 C 语言编译器，那么只要编译器没有 bug，就可以说“编译器接受的输入是正确的 C 程序，编译器不接受的输入是非法的 C 程序”。也就是说，只要有普通计算机的能力，“是否符合 C 的语法”这个问题就是可判定的；编译器整体上可以说是一种比上下文无关文法更强的语法判定器。像这样，总是能够用 YES/NO 判定是否符合语法的文法，称为 `Decidable`。

也可以考虑不是 `Decidable` 的文法。例如，“给定一个计算机程序作为输入并执行它时，该程序最终会执行 `exit` 函数结束，还是会无限执行下去”这个问题，已经被证明通常无法在不实际执行程序的情况下判定 YES/NO（这里假定在内存无限的虚拟计算机上执行）。也就是说，对于程序是否会停止这个问题，当程序会停止时可以回答 YES；但如果程序不会停止，就只会无限执行下去，无法回答 NO。像这样，判定器不仅可能返回 YES/NO，也可能永远运行不结束的文法类别，称为 `Turing-recognizable`。

也就是说，这里存在一个语法层级：正则表达式 < 上下文无关文法 < `Decidable` < `Turing-recognizable`。这类语法层级作为计算机科学的一部分被广泛研究。著名的未解决问题 P≟NP 也与语法层级有关。

## 1973 年的 C 编译器

到目前为止，我们一直在增量式地构建编译器。这个开发过程从某种意义上可以说是在沿着 C 的历史前进。

观察现在的 C，会发现不少意义不太明确或不必要地复杂的地方，但这些东西离开历史就无法理解。现在 C 中一些令人费解之处，如果阅读早期 C 的代码，看看早期 C 的形态以及后来语言和编译器的发展过程，就会有很多地方变得说得通。

C 作为 Unix 的语言，从 1972 年开始开发。1972 年或 1973 年左右，也就是 C 历史中极早期的源代码保存在磁带中，从中读取出的文件已经公开在互联网上。我们稍微看一下当时 C 编译器的代码。下面展示的是一个函数，它以 `printf` 格式接收消息，并把它作为编译错误消息显示出来。

```c
error(s, p1, p2) {
  extern printf, line, fout, flush, putchar, nerror;
  int f;

  nerror++;
  flush();
  f = fout;
  fout = 1;
  printf("%d: ", line);
  printf(s, p1, p2);
  putchar('\n');
  fout = f;
}
```

它看起来有些奇妙，像 C 又不像 C。当时的 C 就是这样的语言。读这段代码首先会注意到，和我们正在构建的编译器早期阶段一样，函数返回值和参数都没有类型。这里的 `s` 应该是指向字符串的指针，`p1` 和 `p2` 应该是整数，但在当时的机器上所有东西大小都一样，所以变量就像这样没有类型。

第 2 行写着 `error` 引用的全局变量和函数的声明。当时的 C 编译器还没有头文件，也没有 C preprocessor，所以程序员需要像这样告诉编译器变量和函数的存在。

和我们当前的编译器一样，函数只检查名字是否存在，并不检查参数类型或参数数量是否一致。只要把预期数量的参数压到栈上，然后直接跳转到函数本体，函数调用就会成功，所以当时大概认为这样就可以了。

`fout` 是保存输出目标 file descriptor 编号的全局变量。这个时候还没有 `fprintf`，如果要把字符串写到标准错误输出而不是标准输出，就需要通过全局变量切换输出目标。

`error` 中调用了两次 `printf`。第二次 `printf` 除了格式字符串之外，还传入了两个值。那么，如果要显示只接收一个值的错误消息，当时是怎么做的呢？

实际上，即使用较少的参数强行调用这个 `error` 函数，它也能正常工作。请回想一下，这个阶段还不存在函数参数检查。`s`、`p1`、`p2` 这些参数只是分别指向从栈指针开始的第 1、2、3 个 word，编译器并不关心实际是否传入了相当于 `p2` 的值。`printf` 只会按照第一个参数字符串中包含的 `%d`、`%s` 个数访问额外参数，所以如果消息只包含一个 `%d`，`p2` 完全不会被访问。因此，即使参数个数不一致也没有问题。

像这样，早期的 C 编译器和当前阶段的 9cc 有很多相似之处。

再看一个代码示例。下面的代码是一个函数，它把传入的字符串复制到静态分配的区域，并返回指向该区域开头的指针。也就是说，这是一个使用静态区域的、类似 `strdup` 的函数。

```c
copy(s)
char s[]; {
  extern tsp;
  char tsp[], otsp[];

  otsp = tsp;
  while(*tsp++ = *s++);
  return(otsp);
}
```

当时还没有发明 `int *p` 这种形式的声明语法。取而代之，指针类型会写成 `int p[]` 这样的声明。函数参数列表和函数本体之间插入了类似变量定义的东西，这是为了把 `s` 声明为指针类型。

这个早期 C 编译器还有其他值得特别说明的地方。

- 在这个时间点，结构体还不存在。

- `&&` 和 `||` 这样的运算符也还没有。当时 `&` 和 `|` 只有在 `if` 等条件表达式中才会成为逻辑运算符，这是一种依赖上下文的行为。

- 像 `+=` 这样的运算符当时写作 `=+`。这种语法存在一个问题：如果本来想给 `i` 赋值 `-1`，但没有加空格写成 `i=-1`，就会被解释成 `i =- 1`，导致 `i` 被递减，产生意外行为。

- 整数类型只有 `char` 和 `int`，还没有 `short` 或 `long`。也不存在声明“函数指针数组”这类类型的语法，无法描述复杂类型。

除了上面这些，70 年代初期的 C 还缺少各种功能。即便如此，从上面的源代码可以看出，这个 C 编译器是用 C 写成的。甚至在还没有结构体的时代，C 就已经能够自举了。

观察旧源代码，也可以推测 C 的一部分难懂语法为什么会变成现在这种形态。如果语法规定 `extern`、`auto`、`int` 或 `char` 后面一定跟变量名，那么变量定义的解析就很简单。表示指针的 `[]` 如果也只是紧跟在变量名后面，那么解析也很简单。不过，如果沿着这个早期编译器中可以看到的方向继续发展这种语法，就会逐渐变成今天这种不必要地复杂的形态，这似乎也可以理解。

1973 年前后，Unix 和 C 的共同开发者 Dennis Ritchie 所做的正是增量式开发。他在发展 C 语言本身的同时，用 C 编写它的编译器。现在的 C 并不是语言功能不断增加后到达某个特殊节点的完成形，而只是 Dennis Ritchie 在某个时间点认为“语言功能已经足够”时，它就作为语言完成了。

我们的编译器也没有从一开始就追求完成形。C 的完成形本身并没有特别意义，所以也没有太大必要特别追求它。始终把语言作为“在当前时点拥有合理功能集合的语言”继续开发，并最终让它变成 C，这正是原始 C 编译器采用过的、有来历的开发方法。放心继续开发吧！

#### 专栏：Rob Pike 的 5 条编程规则

9cc 受到了 Rob Pike 编程观念的影响。Rob Pike 是 C 作者 Dennis Ritchie 以前的同事，是 Go 语言的作者，也是与 Unix 作者 Ken Thompson 一起开发 Unicode 的 UTF-8 的人。

下面引用 Rob Pike 的“编程的 5 条规则”（Rob Pike's 5 Rules of Programming）。

1. 无法预测程序的哪一部分会消耗时间。瓶颈会出现在令人意外的地方，所以在知道瓶颈在哪里之前，不要随意预测瓶颈位置并加入性能 hack。

2. 要测量。在测量之前不要试图优化。而且即使测量了，也不要优化代码中除极端缓慢部分以外的地方。

3. 精巧的算法在 `n` 很小时很慢，而 `n` 通常很小。精巧算法的常数项很大。除非知道 `n` 通常很大，否则不要追求精巧。（即使 `n` 很大，也先应用规则 2。）

4. 精巧算法比简单算法更容易出 bug，也更难实现。应该使用简单的算法和数据结构。

5. 数据才重要。如果选择正确的数据结构，并能很好地表示数据，算法几乎总会变得显而易见。编程的中心不是算法，而是数据结构。

## 步骤 12：加入控制语法

之后的章节还在撰写中。到目前为止的章节，作者认为已经尽量写得细致；但从这里开始，坦率地说还没有达到公开发布的水准。不过，能读到这里的人，应该也可以自己补足必要内容继续阅读；也有人可能只是想要一个“接下来该按什么步骤推进”的路标。因此从这个意义上，作者仍然把这些内容公开出来。

本节会向语言中加入 `if`、`if ... else`、`while`、`for` 这样的控制结构。这些控制结构乍看起来复杂，但如果直接按朴素方式编译到汇编，实现相对简单。

汇编中不存在与 C 的控制结构直接对应的东西，所以 C 的控制结构在汇编中会用分支指令和标签表示。这在某种意义上等同于把控制结构改写成使用 `goto` 的形式。既然人可以手工把控制结构改写成 `goto` 语句，那么编译器只要按模式生成代码，也就能自然地实现控制结构。

控制语法还包括 `do ... while`、`goto`、`continue`、`break` 等各种语法，但这些在当前阶段还不需要实现。

下面展示加入 `if`、`while`、`for` 后的新语法。

```ebnf
program = stmt*
stmt    = expr ";"
        | "if" "(" expr ")" stmt ("else" stmt)?
        | "while" "(" expr ")" stmt
        | "for" "(" expr? ";" expr? ";" expr? ")" stmt
        | ...
...
```

读取 `expr? ";"` 时，可以向前看 1 个 token。如果下一个 token 是 `;`，就认为 `expr` 不存在；否则读取 `expr`。

`if (A) B` 会编译成下面这样的汇编。

```asm
  编译 A 的代码 // 结果应该位于 stack top
  pop rax
  cmp rax, 0
  je .LendXXX
  编译 B 的代码
.LendXXX:
```

也就是说，`if (A) B` 会被展开成类似下面的形式。

```c
if (A == 0)
  goto end;
B;
end:
```

`XXX` 请使用连续编号等方式，确保所有标签都是唯一的。

`if (A) B else C` 会编译成下面这样的汇编。

```asm
  编译 A 的代码 // 结果应该位于 stack top
  pop rax
  cmp rax, 0
  je .LelseXXX
  编译 B 的代码
  jmp .LendXXX
.LelseXXX:
  编译 C 的代码
.LendXXX:
```

也就是说，`if (A) B else C` 会展开成下面这样。

```c
if (A == 0)
  goto els;
B;
goto end;
els:
C;
end:
```

读取 `if` 语句时，向前看 1 个 token，检查是否有 `else`。如果有 `else`，就作为 `if ... else` 编译；如果没有，就作为不带 `else` 的 `if` 编译。

`while (A) B` 会编译成下面这样。

```asm
.LbeginXXX:
  编译 A 的代码
  pop rax
  cmp rax, 0
  je .LendXXX
  编译 B 的代码
  jmp .LbeginXXX
.LendXXX:
```

也就是说，`while (A) B` 会展开成类似下面的代码。

```c
begin:
if (A == 0)
  goto end;
B;
goto begin;
end:
```

`for (A; B; C) D` 会编译成下面这样。

```asm
  编译 A 的代码
.LbeginXXX:
  编译 B 的代码
  pop rax
  cmp rax, 0
  je .LendXXX
  编译 D 的代码
  编译 C 的代码
  jmp .LbeginXXX
.LendXXX:
```

与 `for (A; B; C) D` 对应的 C 代码如下。

```c
A;
begin:
if (B == 0)
  goto end;
D;
C;
goto begin;
end:
```

另外，以 `.L` 开头的标签是 assembler 会特殊识别的名字，会自动成为文件作用域。文件作用域的标签可以从同一个文件中引用，但不能从其他文件引用。因此，让编译器为 `if` 或 `for` 生成的标签以 `.L` 开头，就不用担心它们与其他文件中包含的标签冲突。

请用 `cc` 编译小型循环，并参考它输出的汇编来实现。

#### 专栏：编译器辅助的运行时错误检测

用 C 写程序时，经常会把数据写到数组末尾之外，或者因为指针 bug 破坏无关的数据结构。这类 bug 也可能成为安全漏洞，所以有一种思路是借助编译器，在运行时主动检测 bug。

例如给 GCC 传入 `-fstack-protector` 选项后，编译出的函数会在 prologue 中向函数帧写入一个被称为 canary 的指针大小随机整数，并在 epilogue 中确认 canary 的值没有变化。这样一来，如果数组 buffer overflow 导致栈内容在不知不觉中被覆盖，canary 的值几乎一定也会改变，因此可以在函数返回时检测到错误。检测到错误时，程序通常会立即终止。

LLVM 中有 TSan（ThreadSanitizer），可以输出用于在运行时检测多个线程未正确获取锁就访问共享数据结构的代码。另外，LLVM 的 UBSan（UndefinedBehaviorSanitizer）可以输出用于在运行时检测是否不小心触发 C 未定义行为的代码。例如，带符号整数溢出在 C 中是未定义行为，因此发生带符号整数溢出时，UBSan 会报告错误。

TSan 等工具会让程序运行速度变慢数倍，所以不适合作为日常使用程序的编译选项加入；但像 stack canary 这样运行时成本相对较低的功能，在某些环境中会默认开启。

这种借助编译器的动态错误检测，近年来被广泛研究，并且对于使用 C、C++ 这样非内存安全语言编写相对安全的程序作出了很大贡献。

## 步骤 13：block

这个步骤支持“block”，也就是可以在 `{ ... }` 之间写多个 statement 的结构。block 的正式名称是“复合语句”（compound statement），但因为这个词比较长，通常简称为 block。

block 的作用是把多个 statement 汇总成一个 statement。前面步骤中实现的 `if` 和 `while`，只允许条件表达式成立时执行一个 statement；实现 block 后，就能像 C 一样，在那里写由 `{}` 包围的多条语句。

函数体其实也是 block。在语法上，函数体必须是 block。函数定义中的 `{ ... }`，在语法上其实和写在 `if` 或 `while` 后面的 `{ ... }` 是同一种东西。

加入 block 后的语法如下。

```ebnf
program = stmt*
stmt    = expr ";"
        | "{" stmt* "}"
        | ...
...
```

在这个语法中，如果 `stmt` 以 `"{"` 开始，那么在出现 `"}"` 之前可以出现 0 个或多个 `stmt`。要解析 `stmt* "}"`，可以用 `while` 语句反复调用 `stmt`，直到出现 `"}"`，并把结果作为 vector 返回。

为了实现 block，请添加表示 block 的节点类型 `ND_BLOCK`。表示节点的结构体 `Node` 需要增加一个 vector，用来持有 block 中包含的表达式。在 code generator 中，如果节点类型是 `ND_BLOCK`，就按顺序生成该节点中包含的 statement 的代码。注意，每个 statement 都会在栈上留下一个值，所以不要忘记每次都把它 pop 掉。

## 步骤 14：支持函数调用

这个步骤的目标是识别 `foo()` 这样的无参数函数调用，并把它编译为 `call foo`。

下面展示加入函数调用后的新语法。

```ebnf
...
primary = num
        | ident ("(" ")")?
        | "(" expr ")"
```

读取 `ident` 后向前看 1 个 token，就可以区分这个 `ident` 是变量名还是函数名。

测试时，可以准备一个内容类似 `int foo() { printf("OK\n"); }` 的 C 文件，用 `cc -c` 把它编译为 object file，再和自己编译器的输出链接。这样应该就能确认整体可以正确链接，并且自己想调用的函数也被正确调用了。

它能工作之后，接下来请支持 `foo(3, 4)` 这样的函数调用。不需要检查参数个数或类型。只要按顺序求值参数，栈上就会形成应传给函数的参数；然后按照 x86-64 ABI 规定的顺序把它们复制到寄存器，并 `call` 函数即可。不需要支持超过 6 个的参数。

测试时和上面一样，可以适当准备一个类似 `int foo(int x, int y) { printf("%d\n", x + y); }` 的函数，并把它链接进去，就应该能确认行为。

x86-64 的函数调用 ABI（只要采用上面这种做法）并不复杂，但有一个注意点：函数调用前 `RSP` 必须是 16 的倍数。`push` 和 `pop` 会以 8 字节为单位改变 `RSP`，所以发出 `call` 指令时，`RSP` 不一定总是 16 的倍数。如果不遵守这个约定，那些假定 `RSP` 是 16 的倍数的函数，就会出现大约一半概率崩溃的神秘现象。请在调用函数前调整 `RSP`，让 `RSP` 成为 16 的倍数。

## 步骤 15：支持函数定义

完成到这里之后，接下来就让语言能够定义函数。不过 C 的函数定义语法分析比较麻烦，所以不会一开始就全部实现。目前我们的语言中只有 `int` 类型，因此不实现 `int foo(int x, int y) { ... }` 这样的语法，而是实现省略类型名的 `foo(x, y) { ... }` 这种语法。

被调用方需要能够用 `x`、`y` 这样的名字访问参数，但目前无法直接用名字访问通过寄存器传来的值。那么该怎么做呢？可以把代码编译成好像存在名为 `x`、`y` 的局部变量一样，并在函数 prologue 中，把寄存器的值写入栈上为这些局部变量准备的区域。这样之后就不需要特别区分参数和局部变量了。

到目前为止，整体行为相当于代码被隐式包在 `main() { ... }` 中。现在废除这一点，要求所有代码都写在某个函数中。这样在解析 top level 时，首先读取到的 token 必然是函数名，之后必然跟着参数列表，再之后应该就是函数体，因此很容易读取。

完成这个步骤后，就可以递归计算并显示 Fibonacci 数列，应该会一下子变得有趣起来。

## 二进制级别的接口

C 语言规范规定的是源代码级别的规范。例如，语言规范会规定用什么写法可以定义函数，include 哪个文件会声明哪些函数。另一方面，符合标准写出的源代码会被转换成怎样的机器码，并不由语言规范规定。C 语言标准并不是以某个特定指令集为前提制定的，所以这可以说是理所当然的。

因此，机器码级别的规范乍看起来似乎没有必要严格规定，但实际上每个平台都会在一定程度上规定规范。这种规范称为 ABI（Application Binary Interface）。

本书到目前为止说明的函数调用方式中，参数会按特定顺序放入寄存器。另外，返回值会设置到 `RAX`。这类函数调用方式的规则称为“函数调用约定”（function calling convention）。函数调用约定是 ABI 的一部分。

C 语言的 ABI 除了参数和返回值的传递方式，还包括下面这些内容。

- 函数调用中会被改变和不会被改变的寄存器（`RBP` 等需要在返回前恢复原值，但有些寄存器不需要恢复原值）

- `int`、`long` 等类型的大小

- 结构体布局规则（结构体成员在内存中实际如何排列的规则）

- bit-field 的布局规则（例如 bit-field 是从最低位开始排列，还是从最高位开始排列）

ABI 在软件层面上可以说只是一种约定，因此可以设想与本书说明不同的 ABI；但不兼容 ABI 的代码无法相互调用和使用，所以基本上由 CPU 厂商或 OS 厂商定义平台标准 ABI。在 x86-64 上，广泛使用的有两种：Unix 和 macOS 使用的 System V ABI，以及 Windows 使用的 Microsoft ABI。注意，这两种调用约定并不是因为存在必然性才分开，只是不同的人分别制定了不同约定。

本书到目前为止已经做过从自制编译器调用由其他编译器编译出的函数这样的事情。之所以能够这样做，是因为我们的 C 编译器和其他编译器使用的是相同 ABI。
