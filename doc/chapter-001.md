# 前言

这本在线书仍在编写中，并不是完成版。反馈表单。

这本书塞进了不少内容，作为一本书来说可以说有些“贪心”。本书会制作一个把 C 语言源代码转换为汇编语言的程序，也就是 C 编译器。编译器本身也会用 C 来开发。当前目标是实现 self-host，也就是让自制编译器能够编译它自己的源代码。

为了避免编译器相关说明的难度突然升得太高，本书会在全书范围内逐步深入各种主题。理由如下。

从概念上说，编译器可以分成语法分析、中间处理、代码生成等多个阶段。常见的教科书式写法会为每个主题单独设章讲解，但这种写法容易在中途变得过于狭窄和深入，使读者难以跟上。

另外，如果按阶段逐个开发，在所有阶段完成之前都无法实际运行编译器。因此，直到整体开始工作之前，即使自己的理解或代码存在决定性错误，也很难发现。更根本地说，在亲手做出来之前，往往并不清楚下一个阶段期望什么输入，所以也不清楚前一个阶段应该输出什么。还有一个问题是，在完成之前完全不能编译任何代码，很难维持动力。

为了避开这个陷阱，本书采用了另一种方法。在本书开头，读者会实现一种语言规格极其简单的“自制语言”。这种语言非常简单，因此在实现它的时候，并不需要详细了解编译器的制作方法。之后，读者会在全书过程中不断给这门“自制语言”添加功能，最终把它培养成与 C 一致的语言。

在这种增量式开发方法中，会一边做细小的提交，一边 step by step 地制作编译器。在这种开发方式里，无论哪个提交点，编译器在某种意义上始终都是“完成形”。某个阶段它可能只能做电卓级别的事情，某个阶段可能只是 C 的一个相当受限的子集，某个阶段则可能已经是几乎可以称为 C 的语言。关键在于，在任何阶段都以一种符合当时完成度的、合理规格的语言为目标。开发过程中不会只让某些功能突兀地变得很像 C。

关于数据结构、算法以及计算机科学方面的知识，也会根据开发阶段依次说明。

通过增量开发，在阅读本书的任意时点，读者都能均衡地掌握“到当前水平为止，如何制作一门合理语言”的知识。这比只对编译器制作中的一部分主题极端深入要好得多。等读完本书时，读者应该已经对所有主题都获得了较为均衡的知识。

同时，这本书也是一本说明如何从零开始编写大型程序的书。制作大型程序的技能与学习数据结构和算法不同，是一种独特的技能，但讲解这类技能的书似乎并不多。而且，即使有人讲解，如果不亲自体验，也很难真正理解开发方法的好坏。本书被设计成让“把自制语言培养成 C 语言”的过程本身成为一次良好开发方法的实践体验。

如果作者的设想成功，读者通过阅读本书，不仅能学到编译器制作技巧和 CPU 指令集知识，还应能学到如何把大型程序拆成小步骤逐步制作、软件测试的方法、版本管理的方法，甚至学到面对编译器制作这类有野心的项目时应有的心态。

本书的目标读者是普通的 C 程序员。不需要是精通 C 语言规格的超级 C 程序员。只要理解指针和数组，并且至少在花一些时间的情况下能够读懂别人写的小规模 C 程序，就足够了。

在写作本书时，关于语言规格和 CPU 规格，作者不仅会单纯说明规格，还尽可能解释为什么会选择这样的设计。同时，书中也穿插了关于编译器、CPU、计算机行业及其历史的专栏，希望读者能带着兴趣愉快地读下去。

制作编译器是一件非常有趣的事。最开始，自制语言只能做一些简单到近乎可笑的事情；但随着开发推进，它很快就会成长得连自己都惊讶，越来越像 C 语言，并且像魔法一样顺利运行。实际开发时，经常会遇到这样的惊喜：某段较大的测试代码在当时看来似乎不可能被顺利编译，却能够无错误地编译，并且完全正确地运行。即便去看这类代码编译出来的汇编，自己也未必能马上理解。有时甚至会感觉自制编译器仿佛拥有了超越作者本人的智能。编译器是一种即使理解其机制，也仍会让人隐约觉得“为什么能运行得这么好”的神奇程序。你也一定会沉迷于它的魅力。

那么，前言就到这里。现在就和作者一起跳进编译器开发的世界吧！

## 专栏：为什么是 C 语言

在众多编程语言中，本书为什么选择 C 呢？或者说，为什么不选择自制语言呢？关于这一点，并不是说必须非 C 不可；但如果为了学习输出 native code 的编译器制作方法而必须选择某种语言，C 应该是为数不多的合理选择之一。

解释器方式的语言很难让人学到太多低层知识。另一方面，C 通常会被编译成汇编，因此通过制作编译器，可以在学习 C 本身的同时，也学习 CPU 指令集以及程序运行机制等内容。

C 使用广泛，所以当编译器能够正常工作之后，可以下载第三方源代码来编译着玩。比如可以构建迷你 Unix xv6 来尝试。如果编译器的完成度足够高，甚至应该能够编译 Linux kernel。这样的玩法在小众语言或自制语言中是做不到的。

像 C 这样会被编译为 native 机器码的静态类型语言中，至少与 C 同样广泛使用的还有 C++。但是 C++ 的语言规格过于庞大，不可能轻松制作自制编译器，因此现实中不能作为选择。

设计并实现原创语言，对于磨炼语言设计感觉来说是有好处的，但也存在陷阱。实现起来麻烦的地方，可以通过在语言规格中避开它来逃避实现。像 C 这样语言规格作为标准已经给定的语言则不能这样做。从学习角度看，这种约束反而相当有益。

## 本书的记法

函数、表达式、命令等会在正文中以等宽字体显示，例如 `main`、`foo=3`、`make`。

跨多行的代码会像下面这样用等宽字体显示在框中。

```c
int main() {
  printf("Hello world!\n");
  return 0;
}
```

如果框中的代码是希望用户直接输入的 shell 命令，那么以 `$` 开头的行表示提示符。请把该行 `$` 之后的内容输入到 shell 中（不要输入 `$` 本身）。不以 `$` 开头的行表示输入命令后的输出。例如下面的块，是用户输入 `make` 并按下回车时的执行示例。`make` 命令的输出是 `make: Nothing to be done for \`all'.`。

```console
$ make
make: Nothing to be done for `all'.
```

## 本书假定的开发环境

本书假定的是运行在 Intel、AMD 等所谓普通 PC 上的 64 位 Linux 环境。请读者根据自己使用的发行版，预先安装 `gcc`、`make` 等开发工具。如果是 Ubuntu，可以执行下面的命令安装本书使用的命令。

```console
$ sudo apt update
$ sudo apt install -y gcc make git binutils libc6-dev
```

macOS 与 Linux 在汇编源代码层面有相当高的兼容性，但并不是完全兼容（具体来说，不支持“静态链接”这一功能）。按照本书内容制作支持 macOS 的 C 编译器并非不可能，但实际尝试时，应该会在各种细节上被不兼容问题困扰。同时学习 C 编译器制作技巧以及 macOS 与 Linux 的差异并不值得推荐。因为当某些东西不能顺利工作时，你会很难判断到底是哪一边的理解错了。

因此，本书不把 macOS 作为目标环境。在 macOS 上，请使用某种虚拟环境准备 Linux 环境。如果读者是第一次准备 Linux 虚拟环境，可以参考附录 3，其中总结了使用 Docker 创建开发环境的方法。

Windows 与 Linux 在汇编源代码层面不兼容。不过，在 Windows 10 中，可以把 Linux 像一个应用程序一样运行在 Windows 上，借助它就可以在 Windows 上继续开发。这个 Linux 兼容环境就是 Windows Subsystem for Linux（WSL）。如果要在 Windows 上实践本书内容，请安装 WSL，并在其中进行开发。

## 专栏：交叉编译器

运行编译器的机器称为“host”，编译器输出的代码所运行的机器称为“target”。本书中二者都是 64 位 Linux 环境，但 host 和 target 不一定必须相同。

host 与 target 不同的编译器称为交叉编译器。例如，一个运行在 Windows 上、用于生成 Raspberry Pi 可执行文件的编译器，就是交叉编译器。交叉编译器常用于 target 机器性能不足以运行编译器，或 target 机器比较特殊的情况。

## 专栏：编译编译器的编译器

C 编译器由 C 编写这类自引用状况并不罕见。除 C 以外，也有许多语言实现是用该语言自身编写的。

如果已经存在语言 X 的实现，那么使用这门语言自身制作新的 X 编译器，在逻辑上并不矛盾。如果想要 self-host，只需要先用已有编译器推进开发，等自制编译器完成后再切换过去即可。本书中我们要做的正是这种方法。

但是，如果没有已有编译器，应该怎么办呢？这时只能使用另一门语言来编写。当打算实现 self-host，并编写语言 X 的第一个编译器时，需要先用不同于 X 的既有语言 Y 来写；等编译器的完成度提高之后，再把编译器自身从 Y 语言改写成 X 语言。

现代复杂编程语言的编译器，如果沿着“用于编译该语言实现的另一个编译器”这一谱系不断追溯，最终应该会抵达计算机黎明期某个人直接用机器语言编写的简单汇编器。这个汇编器在某种意义上是现存所有语言实现的终极祖先，至于它是单一的还是多个的，我们并不知道；但可以肯定，如今的编译器是从极少数祖先出发发展而来的。编译器以外的可执行文件通常也是由编译器生成的文件，所以现存几乎所有可执行文件，都是那个原始汇编器的间接后代。这就像生命起源一样，是很有趣的话题。
