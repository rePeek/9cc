# 制作计算器级语言

本章会作为制作 C 编译器的第一步，支持四则运算以及其他算术运算符，使编译器能够编译下面这样的表达式。

```text
30 + (4 - 2) * -5
```

这看起来像是一个很简单的目标，但实际上相当困难。数学表达式中存在一些结构，例如括号中的表达式优先、乘法比加法优先等；如果不能以某种方式理解这些结构，就无法正确计算。但是，作为输入给出的数学表达式只是一串扁平的字符，并不是结构化数据。为了正确求值表达式，需要解析字符序列，并很好地导出其中隐藏的结构。

如果没有任何前置知识就去解决这种语法分析问题，会相当困难。实际上，这类问题过去曾被认为是难题，尤其在 1950 年代到 1970 年代，人们进行了大量研究，并开发出了各种算法。得益于这些成果，如今只要知道方法，语法分析已经不再是特别困难的问题。

本章会说明语法分析中最常见的算法之一：“递归下降语法分析法”（recursive descent parsing）。GCC、Clang 等大家日常使用的 C/C++ 编译器，也使用递归下降语法分析法。

不限于编译器，在编程中经常会遇到读取某种结构化文本的需求。本章学习的技巧也可以直接用于这些问题。本章学习的语法分析方法，毫不夸张地说，是可以用一辈子的技术。请通过阅读本章理解算法，把语法分析这项技能放进自己的程序员工具箱中。

## 步骤 1：制作只能编译一个整数的语言

请思考最简单的 C 语言子集会是什么样的。读者会想象出怎样的语言呢？是只有 `main` 函数的语言吗？或者是只由一个表达式组成的语言吗？追究到底，只由一个整数构成的语言，应该可以说是能想到的最简单子集。

在这个步骤中，我们先来实现这个最简单的语言。

这个步骤要创建的程序，是一个从输入读取一个数字，并输出把该数字作为程序退出码退出的汇编的编译器。也就是说，输入只是像 `42` 这样的字符串，读取它后，编译器会输出如下汇编。

```asm
.intel_syntax noprefix
.globl main

main:
        mov rax, 42
        ret
```

`.intel_syntax noprefix` 是一个汇编器命令，用来在多种汇编写法中选择本书使用的 Intel 记法。在这次制作的编译器中，请约定总是在开头放入这一行。其他行与前一章说明的一样。

读者在这里可能会想：“这种程序不能算编译器吧。”坦白说，作者也这么觉得。但是，这个程序接收一种只由一个数值组成的语言作为输入，并输出与该数值对应的代码；从定义上说，它就是一个合格的编译器。这样简单的程序经过改造后，很快就能做相当复杂的事情，所以先完成这个步骤吧。

实际上，从整个开发流程来看，这个步骤非常重要。因为之后的开发会以这个步骤制作的东西作为骨架继续推进。在这个步骤中，除了制作编译器本体，还会创建构建文件（Makefile）、自动测试，并设置 git 仓库。下面逐一来看这些工作。

本书制作的 C 编译器名为 `9cc`。`cc` 是 C compiler 的缩写。数字 `9` 没有什么特别含义，只是因为作者以前制作的 C 编译器名为 `8cc`，所以作为下一部作品取名为 `9cc`。当然，大家可以取自己喜欢的名字。不过，不要在开始前过度纠结命名，以至于无法开始制作编译器。包括 GitHub 仓库在内，名字之后都可以改，所以用一个合适的临时名字开始也没有问题。

### 专栏：Intel 记法与 AT&T 记法

除了本书使用的 Intel 记法之外，还有一种名为 AT&T 记法的汇编器记法，它也以 Unix 为中心被广泛使用。`gcc` 和 `objdump` 默认会以 AT&T 记法输出汇编。

在 AT&T 记法中，结果寄存器位于第 2 个参数。因此，对于双参数指令，参数要按相反顺序书写。寄存器名要加 `%` 前缀，例如 `%rax`。数值要加 `$` 前缀，例如 `$42`。

另外，引用内存时，不使用 `[]`，而是使用 `()`，并以一种独特记法书写表达式。下面给出几个对比例子。

```asm
mov rbp, rsp   // Intel
mov %rsp, %rbp // AT&T

mov rax, 8     // Intel
mov $8, %rax   // AT&T

mov [rbp + rcx * 4 - 8], rax // Intel
mov %rax, -8(rbp, rcx, 4)    // AT&T
```

这次制作的编译器考虑到可读性，决定使用 Intel 记法。Intel 的指令集手册使用 Intel 记法，因此也有一个优点：可以直接把手册中的描述写成代码。AT&T 记法和 Intel 记法的表达能力相同。无论使用哪种记法，生成的机器码指令序列都是一样的。

### 制作编译器本体

通常会把文件作为输入传给编译器，但这里打开并读取文件比较麻烦，所以直接把代码作为命令的第 1 个参数传入。读取第 1 个参数作为数值，并把它嵌入固定格式汇编中的 C 程序，可以像下面这样简单编写。

```c
#include <stdio.h>
#include <stdlib.h>

int main(int argc, char **argv) {
    if (argc != 2) {
      fprintf(stderr, "参数个数不正确\n");
      return 1;
    }

    printf(".intel_syntax noprefix\n");
    printf(".globl main\n");
    printf("main:\n");
    printf(" mov rax, %d\n", atoi(argv[1]));
    printf(" ret\n");
    return 0;
}
```

创建一个名为 `9cc` 的空目录，在其中用上述内容创建 `9cc.c` 文件。然后像下面这样运行 `9cc`，确认它的行为。

```console
$ cc -o 9cc 9cc.c
$ ./9cc 123 > tmp.s
```

第 1 行编译 `9cc.c`，创建名为 `9cc` 的可执行文件。第 2 行把 `123` 这个输入传给 `9cc`，生成汇编，并写入名为 `tmp.s` 的文件。来确认 `tmp.s` 的内容。

```console
$ cat tmp.s
.intel_syntax noprefix
.globl main
main:
  mov rax, 123
  ret
```

如你所见，它已经顺利生成了。把这样生成的汇编文件交给汇编器，就可以创建可执行文件。

在 Unix 中，`cc`（或 `gcc`）不仅是 C 和 C++，也被视为许多语言的前端，它会根据给定文件的扩展名判断语言，并启动编译器或汇编器。因此这里和编译 `9cc` 时一样，把扩展名为 `.s` 的汇编文件传给 `cc`，就可以进行汇编。下面是执行汇编并运行生成的可执行文件的例子。

```console
$ cc -o tmp tmp.s
$ ./tmp
$ echo $?
123
```

在 shell 中，可以通过 `$?` 变量访问上一条命令的退出码。上面的例子显示了与传给 `9cc` 的参数相同的数字 `123`。也就是说，它正常工作。请试着传入 0 到 255 范围内除 123 之外的其他数字（Unix 的进程退出码范围是 0 到 255），实际确认 `9cc` 能正常工作。

### 创建自动测试

很多读者在业余编程中可能没有写过测试，但本书中，每次扩展编译器时，都会为新代码编写测试。刚开始写测试可能会觉得麻烦，但很快就会明白测试的价值。如果不写测试代码，最终只能每次手动执行类似的测试来确认行为，而手动操作要麻烦得多。

“写测试很麻烦”这种印象，很大一部分可能来自测试框架过于隆重，或者测试思想有时显得教条。例如 JUnit 这样的测试框架有许多方便功能，但导入它、学习用法都需要成本。因此本章不会引入这类测试框架。取而代之，我们会用 shell script 手写一个非常简单的“测试框架”，并用它来写测试。

下面展示测试用 shell script `test.sh`。shell 函数 `assert` 接收两个参数：输入值和期望输出值。它会实际汇编 `9cc` 的结果，并比较实际结果与期望值。这个 shell script 在定义 `assert` 函数后，用它确认 `0` 和 `42` 都能被正确编译。

```bash
#!/bin/bash
assert() {
  expected="$1"
  input="$2"

  ./9cc "$input" > tmp.s
  cc -o tmp tmp.s
  ./tmp
  actual="$?"

  if [ "$actual" = "$expected" ]; then
    echo "$input => $actual"
  else
    echo "$input => $expected expected, but got $actual"
    exit 1
  fi
}

assert 0 0
assert 42 42

echo OK
```

用上述内容创建 `test.sh`，并执行 `chmod a+x test.sh` 让它变为可执行文件。实际运行一下 `test.sh`。如果没有发生错误，`test.sh` 会像下面这样最后显示 `OK` 并结束。

```console
$ ./test.sh
0 => 0
42 => 42
OK
```

如果发生错误，`test.sh` 不会显示 `OK`。取而代之，`test.sh` 会像下面这样显示失败测试中预期的值和实际的值。

```console
$ ./test.sh
0 => 0
42 expected, but got 123
```

想调试测试脚本时，请给 `bash` 加上 `-x` 选项来执行脚本。加上 `-x` 选项后，`bash` 会像下面这样显示执行跟踪。

```console
$ bash -x test.sh
+ assert 0 0
+ expected=0
+ input=0
+ cc -o 9cc 9cc.c
+ ./9cc 0
+ cc -o tmp tmp.s
+ ./tmp
+ actual=0
+ '[' 0 '!=' 0 ']'
+ assert 42 42
+ expected=42
+ input=42
+ cc -o 9cc 9cc.c
+ ./9cc 42
+ cc -o tmp tmp.s
+ ./tmp
+ actual=42
+ '[' 42 '!=' 42 ']'
+ echo OK
OK
```

本书通篇使用的“测试框架”，只是上述这样的 shell script。与 JUnit 等正式测试框架相比，这个脚本可能显得过于简单，但这个 shell script 的简单程度与 `9cc` 自身的简单程度是平衡的，因此这么简单反而更合适。所谓自动测试，本质上只需要能一键运行自己写的代码，并机械地比较结果即可。不要想得太复杂，首先开始测试才是重要的。

### 使用 make 构建

在本书中，读者可能会构建 `9cc` 数百次甚至数千次。创建 `9cc` 的可执行文件，然后运行测试脚本，这些工作每次都一样，所以交给工具会很方便。标准上用于这类用途的就是 `make` 命令。

`make` 执行时，会读取当前目录中名为 `Makefile` 的文件，并执行其中写着的命令。`Makefile` 由以冒号结尾的规则，以及该规则对应的一组命令组成。下面的 `Makefile` 用于自动化这个步骤中想执行的命令。

```makefile
CFLAGS=-std=c11 -g -static

9cc: 9cc.c

test: 9cc
        ./test.sh

clean:
         rm -f 9cc *.o *~ tmp*

.PHONY: test clean
```

请把上述文件以 `Makefile` 这个文件名创建在与 `9cc.c` 相同的目录中。这样一来，只执行 `make` 就能创建 `9cc`，执行 `make test` 就能运行测试。`make` 能理解文件依赖关系，所以修改 `9cc.c` 之后，在执行 `make test` 之前不需要先执行 `make`。只有当名为 `9cc` 的可执行文件比 `9cc.c` 更旧时，`make` 才会在运行测试之前构建 `9cc`。

`make clean` 是删除临时文件的规则。临时文件也可以手动用 `rm` 删除，但如果误删了不想删的文件会很麻烦，所以这种工具性质的操作也写进 `Makefile` 中。

另外，书写 `Makefile` 时需要注意，`Makefile` 的缩进必须是 tab 字符。4 个或 8 个空格都会出错。这只是一个不太好用的语法，但 `make` 是 1970 年代开发的老工具，传统上就是这样。

请务必给 `cc` 传入 `-static` 选项。这个选项会在“动态链接”一章中说明。现在不需要特别考虑这个选项的含义。

## 步骤 2：制作能够处理加减法的编译器

在这个步骤中，我们会扩展上一步创建的编译器，使它不仅能接收 `42` 这样的值，还能接收 `2+11`、`5+20-4` 这样包含加减法的表达式。

像 `5+20-4` 这样的表达式，也可以在编译时计算，并把结果数值（这个例子中是 21）嵌入汇编；但这样就更像解释器而不是编译器了。因此，我们需要输出在运行时执行加减法的汇编。执行加法和减法的汇编指令是 `add` 和 `sub`。`add` 接收两个寄存器，把它们的内容相加，并把结果写入第 1 个参数的寄存器。`sub` 与 `add` 类似，但执行减法。使用这些指令，`5+20-4` 可以编译成如下内容。

```asm
.intel_syntax noprefix
.globl main

main:
        mov rax, 5
        add rax, 20
        sub rax, 4
        ret
```

在上面的汇编中，先用 `mov` 把 5 设置到 RAX，然后给 RAX 加上 20，再减去 4。执行 `ret` 时，RAX 的值应该是 `5+20-4`，也就是 21。我们运行确认一下。把上述文件保存为 `tmp.s`，汇编后执行。

```console
$ cc -o tmp tmp.s
$ ./tmp
$ echo $?
21
```

如上所示，正确显示了 21。

那么，这个汇编文件应该如何创建呢？如果把这种包含加减法的表达式当作一门“语言”来看，这门语言可以定义如下。

- 最开始有一个数字
- 之后跟着 0 个或多个“项”
- 所谓项，就是 `+` 后面跟着数字，或者 `-` 后面跟着数字

把这个定义直接落到 C 代码中，会得到下面这样的程序。

```c
#include <stdio.h>
#include <stdlib.h>

int main(int argc, char **argv) {
 if (argc != 2) {
   fprintf(stderr, "参数个数不正确\n");
   return 1;
 }

 char *p = argv[1];

 printf(".intel_syntax noprefix\n");
 printf(".globl main\n");
 printf("main:\n");
 printf(" mov rax, %ld\n", strtol(p, &p, 10));

 while (*p) {
   if (*p == '+') {
     p++;
     printf(" add rax, %ld\n", strtol(p, &p, 10));
     continue;
   }

   if (*p == '-') {
     p++;
     printf(" sub rax, %ld\n", strtol(p, &p, 10));
     continue;
   }

   fprintf(stderr, "意外字符: '%c'\n", *p);
   return 1;
 }

 printf(" ret\n");
 return 0;
}
```

程序稍微变长了，但前半部分和 `ret` 那一行与以前相同。中间添加了用于读取项的代码。这次程序不只是读取一个数字，所以读取数字之后，必须知道读到了哪里。`atoi` 不会返回读取了多少个字符，因此用 `atoi` 就不知道下一项应该从哪里开始读。所以这里使用了 C 标准库函数 `strtol`。

`strtol` 读取数值后，会更新第 2 个参数的指针，使其指向最后读取字符的下一个字符。因此，读取一个数值之后，如果下一个字符是 `+` 或 `-`，`p` 应该就指向那个字符。上面的程序利用这一点，在 `while` 循环中依次读取项，每读取一个项就输出一行汇编。

接下来马上运行这个改造版编译器。更新 `9cc.c` 文件后，只要执行 `make` 就能创建新的 `9cc` 文件。执行例如下。

```console
$ make
$ ./9cc '5+20-4'
.intel_syntax noprefix
.globl main
main:
  mov rax, 5
  add rax, 20
  sub rax, 4
  ret
```

看起来汇编已经顺利输出了。为了测试这个新功能，给 `test.sh` 添加下面这一行测试。

```bash
assert 21 "5+20-4"
```

做到这里后，把目前为止的变更 commit 到 git。为此执行下面的命令。

```console
$ git add test.sh 9cc.c
$ git commit
```

## 步骤 3：引入 tokenizer

上一步创建的编译器有一个缺点：如果输入中包含空白字符，它就会在那一点出错。例如，像下面这样传入包含空格的字符串 `5 - 3`，在试图读取 `+` 或 `-` 的地方会遇到空白字符，导致编译失败。

```console
$ ./9cc '5 - 3' > tmp.s
意外字符: ' '
```

解决这个问题有几种方法。一个显而易见的方法是在读取 `+` 或 `-` 之前跳过空白字符。这种做法并没有特别的问题，但在这个步骤中，我们会用另一种方法解决问题。这个方法就是在读取表达式之前，先把输入切分成单词。

和日语、英语一样，算术表达式和编程语言也可以认为是由单词序列构成的。例如，可以认为 `5+20-4` 由 `5`、`+`、`20`、`-`、`4` 这 5 个单词构成。这种“单词”称为“token”。token 之间的空白字符只是为了分隔 token 而存在，并不是构成单词的一部分。因此，把字符串切分为 token 序列时去掉空白字符，是很自然的做法。把字符串切分为 token 序列称为“tokenize”。

把字符串分成 token 序列还有其他优点。在把表达式分成 token 时，可以对 token 进行分类并附上类型。例如，`+` 和 `-` 如其所见是符号；另一方面，字符串 `123` 表示数值 123。tokenize 时不仅把输入切分成字符串，还解释每个 token，这样在消费 token 序列时需要考虑的事情就会减少。

对于当前能够处理加减法的表达式语法，token 类型有 3 种：`+`、`-`、数值。进一步地，出于编译器实现上的方便，如果定义一个表示 token 序列结束的特殊类型，程序会更简洁（这类似于字符串以 `'\0'` 结尾）。我们把 token 做成用指针连接的链表，以便处理任意长度的输入。

下面列出引入 tokenizer 后改良版编译器的代码，稍微有些长。

```c
#include <ctype.h>
#include <stdarg.h>
#include <stdbool.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

// token 的种类
typedef enum {
  TK_RESERVED, // 保留字
  TK_NUM,      // 整数 token
  TK_EOF,      // 表示输入结束的 token
} TokenKind;

typedef struct Token Token;

// token 类型
struct Token {
  TokenKind kind; // token 类型
  Token *next;   // 下一个输入 token
  int val;       // kind 为 TK_NUM 时的数值
  char *str;     // token 字符串
};

// 当前关注的 token
Token *token;

// 用于报告错误的函数
// 接收与 printf 相同的参数
void error(char *fmt, ...) {
  va_list ap;
  va_start(ap, fmt);
  vfprintf(stderr, fmt, ap);
  fprintf(stderr, "\n");
  exit(1);
}

// 如果下一个 token 是期望的符号，就读取一个 token
// 并返回 true；否则返回 false。
bool consume(char op) {
  if (token->kind != TK_RESERVED || token->str[0] != op)
    return false;
  token = token->next;
  return true;
}

// 如果下一个 token 是期望的符号，就读取一个 token。
// 否则报告错误。
void expect(char op) {
  if (token->kind != TK_RESERVED || token->str[0] != op)
    error("不是 '%c'", op);
  token = token->next;
}

// 如果下一个 token 是数值，就读取一个 token 并返回该数值。
// 否则报告错误。
int expect_number() {
  if (token->kind != TK_NUM)
    error("不是数字");
  int val = token->val;
  token = token->next;
  return val;
}

bool at_eof() {
  return token->kind == TK_EOF;
}

// 创建新 token 并连接到 cur
Token *new_token(TokenKind kind, Token *cur, char *str) {
  Token *tok = calloc(1, sizeof(Token));
  tok->kind = kind;
  tok->str = str;
  cur->next = tok;
  return tok;
}

// tokenize 输入字符串 p 并返回结果
Token *tokenize(char *p) {
  Token head;
  head.next = NULL;
  Token *cur = &head;

  while (*p) {
    // 跳过空白字符
    if (isspace(*p)) {
      p++;
      continue;
    }

    if (*p == '+' || *p == '-') {
      cur = new_token(TK_RESERVED, cur, p++);
      continue;
    }

    if (isdigit(*p)) {
      cur = new_token(TK_NUM, cur, p);
      cur->val = strtol(p, &p, 10);
      continue;
    }

    error("无法 tokenize");
  }

  new_token(TK_EOF, cur, p);
  return head.next;
}

int main(int argc, char **argv) {
  if (argc != 2) {
    error("参数个数不正确");
    return 1;
  }

  // 进行 tokenize
  token = tokenize(argv[1]);

  // 输出汇编前半部分
  printf(".intel_syntax noprefix\n");
  printf(".globl main\n");
  printf("main:\n");

  // 表达式开头必须是数字，因此检查它
  // 并输出最初的 mov 指令
  printf(" mov rax, %d\n", expect_number());

  // 一边消费 `+ <数>` 或 `- <数>` 这样的 token 序列
  // 一边输出汇编
  while (!at_eof()) {
    if (consume('+')) {
      printf(" add rax, %d\n", expect_number());
      continue;
    }

    expect('-');
    printf(" sub rax, %d\n", expect_number());
  }

  printf(" ret\n");
  return 0;
}
```

这段代码大约 150 行，不能说很短，但并没有做特别取巧的事，所以从上往下读应该能读懂。

下面说明一下上面代码中使用的几个编程技巧。

- parser 读取的 token 序列用全局变量 `token` 表示。parser 通过沿着链表形式的 `token` 前进来读取输入。使用这种全局变量的编程风格也许看起来并不漂亮。但实际上，像这里这样把输入 token 序列当成类似标准输入的 stream 来处理，parser 代码往往更容易阅读。因此这里采用了这种风格。
- 直接操作 `token` 的代码被分离到 `consume`、`expect` 等函数中，其他函数不直接操作 `token`。
- `tokenize` 函数会构建链表。构建链表时，如果创建一个 dummy 的 `head` 元素，把新元素依次连接到它后面，最后返回 `head->next`，代码会比较简单。这种方法中分配给 `head` 元素的内存几乎是浪费的，但分配局部变量的成本几乎为零，所以不需要特别在意。
- `calloc` 和 `malloc` 一样是分配内存的函数。与 `malloc` 不同，`calloc` 会把分配到的内存清零。这里为了省去手动清零元素的麻烦，使用了 `calloc`。

## 步骤 4：改进错误消息

到目前为止制作的编译器，在输入语法错误时，只能让人知道某处发生了错误。这个步骤中，我们来改进这个问题。具体来说，要能够显示下面这种直观易懂的错误消息。

```console
$ ./9cc "1+3++" > tmp.s
1+3++
    ^ 不是数字

$ ./9cc "1 + foo + 5" > tmp.s
1 + foo + 5
    ^ 无法 tokenize
```

为了显示这种错误消息，需要在发生错误时知道错误位于输入的第几个字节。为此，我们把整个程序字符串保存在名为 `user_input` 的变量中，并新定义一个错误显示函数，让它接收指向该字符串中某处的指针。代码如下。

```c
// 输入程序
char *user_input;

// 报告错误位置
void error_at(char *loc, char *fmt, ...) {
  va_list ap;
  va_start(ap, fmt);

  int pos = loc - user_input;
  fprintf(stderr, "%s\n", user_input);
  fprintf(stderr, "%*s", pos, " "); // 输出 pos 个空格
  fprintf(stderr, "^ ");
  vfprintf(stderr, fmt, ap);
  fprintf(stderr, "\n");
  exit(1);
}
```

`error_at` 接收的指针，是指向表示整个输入的字符串中间某处的指针。用这个指针减去指向输入开头的指针，就能知道错误位置是输入的第几个字节，因此可以用 `^` 突出标记该位置。

如果是实用级编译器，输入存在错误时的行为也应该编写测试。不过目前错误消息只是为了辅助调试而输出，所以这个阶段不特别写测试也可以。

### 专栏：源代码 formatter

就像日语文章如果在标点等正字法层面错误很多，会让人难以阅读一样，源代码如果缩进奇怪、空格有无不一致，那么在代码内容之前，就已经不能称为整洁代码。对于代码 formatting 这种可以说无关本质的部分，请机械地应用一定规则，注意写出不会分散注意力、易于阅读的代码。

多人开发时，需要讨论并决定采用什么格式；但本书中是一个人开发，所以可以从比较主流的格式中选择自己喜欢的格式。

最近开发的语言中，有些提供了语言官方 formatter，用来消除“选择什么格式”这种偏好会分歧但并非本质的讨论本身。例如 Go 语言有 `gofmt` 命令，使用它可以漂亮地格式化源代码。`gofmt` 没有用于选择格式风格的选项，也就是说只能格式化成唯一的“Go 官方格式”。通过刻意不给选择，Go 完全解决了“格式应该怎样”这个问题。

C 和 C++ 中有名为 `clang-format` 的 formatter，但本书并不是特别想推荐使用这类工具。请尽量不要先写格式奇怪的代码再事后整理，而是一开始就注意写出外观一致的代码。

### 专栏：由缩进错误引发的安全 bug

曾经因为源代码缩进错误，iOS 和 macOS 产生了严重的安全问题。出现 bug 的位置代码如下。

```c
if ((err = ReadyHash(&SSLHashSHA1, &hashCtx)) != 0)
    goto fail;
if ((err = SSLHashSHA1.update(&hashCtx, &clientRandom)) != 0)
    goto fail;

```

读者能看出 bug 在哪里吗？这段代码乍看像是普通代码片段，但仔细看会发现倒数第 2 个 `goto` 语句并没有进入 `if` 语句内部，而是变成了总会执行的 `goto` 语句。

## 语法的描述方法与递归下降语法分析

接下来，我们想把乘除法和用于优先级的括号，也就是 `*`、`/`、`()` 加入语言。为了做到这一点，有一个很大的技术挑战：乘法和除法有必须在表达式中先计算的规则。例如，表达式 `1+2*3` 必须被解释为 `1+(2*3)`，而不能被解释为 `(1+2)*3`。这种“哪个运算符最先结合”的规则称为“运算符优先级”（operator precedence）。

应该如何处理运算符优先级呢？到目前为止制作的编译器，只是从开头读取 token 序列并输出汇编。如果直接照这个方式扩展并加入 `*` 和 `/`，就会把 `1+2*3` 编译成 `(1+2)*3`。

现有编译器当然能够很好地处理运算符优先级。编译器的语法分析非常强大，无论多复杂的代码，只要符合语法，就能正确解释。编译器的这种行为有时甚至让人感觉它具有超越人类的智能能力；但实际上，计算机没有人类那样的文章阅读能力，所以语法分析必然只通过某种机械机制完成。具体来说，它是如何工作的呢？

本节先暂时放下编码，学习语法分析技术。本节会按以下顺序说明语法分析技术。

1. 通过了解 parser 输出的数据结构，先把握最终目标
2. 学习定义语法规则的规则
3. 基于定义语法规则的规则，学习编写 parser 的技巧

### 用树结构表示语法结构

在编程语言 parser 的实现中，输入通常是扁平的 token 序列，输出则是表示嵌套结构的树。本书制作的编译器也遵循这种构成。

C 语言中，`if`、`while` 等语法元素可以嵌套。用树结构表示这些东西，可以说是自然的表示方式。

数学表达式中有一些结构，例如先计算括号里的内容、乘除法先于加减法计算等。这些结构乍看也许不像树，但实际上用树可以非常简单地表示表达式结构。例如，可以认为表达式 `1*(2+3)` 由下面这棵树表示。

```text
        *
      /   \
     1     +
          / \
         2   3
```

如果采用从树的末端依次计算的规则，上面的树表示“用 1 乘以 2+3”这个表达式。也就是说，在这棵树中，`1*(2+3)` 的具体计算顺序由树的形状本身表示。

再看另一个例子。下面的树表示 `7-3-3`。

```text
        -
      /   \
     -     3
   /   \
  7     3
```

在上面的树中，“减法必须从左到右依次计算”这一规则的应用结果，被树的形状明确表示出来。也就是说，上面的树表示 `(7-3)-3 = 1`，而不是 `7-(3-3) = 7`。如果是后者，对应的树会不是向左变深，而是向右变深。

必须从左侧开始计算的运算符称为“左结合”运算符；必须从右侧开始计算的运算符称为“右结合”运算符。在 C 中，除了赋值的 `=` 以外，大多数运算符都被定义为左结合。

在树结构中，通过让树变深，可以表示任意长的表达式。下面这棵树表示 `1*2+3*4*5`。

```text
            +
          /   \
         *     *
       /  \   / \
      1    2 *   5
            / \
           3   4
```

像上面这样的树称为“语法树”（syntax tree）。尤其是，不在树中保留用于分组的括号等冗余元素、尽量紧凑表示的语法树，称为“抽象语法树”（abstract syntax tree，AST）。上面的语法树都可以称为抽象语法树。

抽象语法树是编译器的内部表示，因此可以根据实现方便适当定义。话虽如此，像加法、乘法这样的算术运算符，是对左边和右边两个对象的运算，所以任何编译器中把它们做成二叉树都很自然。另一方面，函数体中的语句等只是按顺序执行且数量可以任意多的东西，用拥有所有子元素的扁平树来表示会更自然。

语法分析的目标是构建抽象语法树。编译器首先进行语法分析，把输入 token 序列转换为抽象语法树，然后再把这棵语法树转换为汇编。

### 用生成规则定义语法

接下来学习编程语言语法的描述方法。编程语言的大部分语法是使用“生成规则”（production rule）定义的。生成规则是递归定义语法的规则。

可以把这种语法想象成由一些规则定义，例如“‘句子’由‘主语’和‘谓语’构成”，“‘名词短语’由‘名词’构成，或者由‘形容词’后接‘名词短语’构成”。这样一来，从“句子”出发，按照规则展开，就可以生成无数个在已定义语法中合法的句子。

反过来，对于已经存在的句子，也可以通过思考与之匹配的展开过程，判断这个字符串具有怎样的结构。

上述想法原本是为自然语言提出的，但它与计算机处理的数据非常契合，因此生成规则被用于编程语言以及计算机中的各种场景。

### 专栏：乔姆斯基的生成语法

提出生成语法这一想法的是语言学家诺姆·乔姆斯基。他的思想对语言学和计算机科学产生了非常大的影响。

根据乔姆斯基的假说，人类之所以能够说话，是因为人脑中天生存在用于获得生成规则的专用回路。也就是说，人类具有获得递归语言规则的能力，因此能够掌握语言。人类以外的动物没有语言获得能力，他认为这是因为其他动物的大脑中不存在获得生成规则的回路。乔姆斯基的主张在假说提出近 60 年后的今天，仍未被证明也未被反驳，但现在仍被认为具有相当说服力。

### 用 BNF 描述生成规则

为了紧凑且易懂地描述生成规则，有一种记法叫 BNF（Backus-Naur form），以及扩展它得到的 EBNF（Extended BNF）。本书会使用 EBNF 说明 C 的语法。本节先说明 BNF，然后说明 EBNF 的扩展部分。

在 BNF 中，每条生成规则都以 `A = α₁α₂...` 的形式表示。这表示符号 `A` 可以展开为 `α₁α₂...`。`α₁α₂...` 是由 0 个以上符号组成的序列，可以同时包含不能进一步展开的符号，以及会进一步展开的符号（出现在某条生成规则左边的符号）。

不能进一步展开的符号称为“终结符”（terminal symbol）；出现在某条生成规则左边、可以继续展开的符号称为“非终结符”（nonterminal symbol）。由这种生成规则定义的语法，一般称为“上下文无关文法”（context free grammar）。

非终结符可以匹配多条生成规则。例如，如果同时存在 `A = α₁` 和 `A = α₂` 两条规则，意思就是 `A` 可以展开为 `α₁` 或 `α₂`。

生成规则的右边可以为空。在这种规则中，左边符号会展开为长度为 0 的符号序列（也就是无）。不过，表示上如果省略右边会让含义难懂，因此这种情况下通常会在右边写上表示“什么都没有”的符号 `ε`（epsilon），这是普通 BNF 的规则。本书也采用这个规则。

字符串用双引号包围，例如 `"foo"`。字符串始终是终结符。

以上是基本 BNF 规则。在 EBNF 中，除了 BNF 规则，还可以使用以下符号简洁地写出复杂规则。

| 写法 | 含义 |
| --- | --- |
| `A*` | `A` 重复 0 次以上 |
| `A?` | `A` 或 `ε` |
| `A | B` | `A` 或 `B` |
| `( ... )` | 分组 |

例如，在 `A = ("fizz" | "buzz")*` 中，`A` 可以展开为 `"fizz"` 或 `"buzz"` 重复 0 次以上的字符串，也就是 `""`、`"fizz"`、`"buzz"`、`"fizzfizz"`、`"fizzbuzz"`、`"buzzfizz"`、`"buzzbuzz"`、`"fizzfizzfizz"`、`"fizzfizzbuzz"`、……中的任意一种。

### 专栏：BNF 与 EBNF

普通的、非 Extended 的 BNF 中没有 `*`、`?`、`|`、`( ... )` 这类简洁记法，但 BNF 能生成的句子和 EBNF 能生成的句子是一样的。因为可以通过下面这样的改写，把 EBNF 转换为 BNF。

| EBNF | 对应的 BNF |
| --- | --- |
| `A = α*` | `A = αA` 与 `A = ε` |
| `A = α?` | `A = α` 与 `A = ε` |
| `A = α | β` | `A = α` 与 `A = β` |
| `A = α (β₁β₂...) γ` | `A = α B γ` 与 `B = β₁β₂...` |

例如，使用 `A = αA` 与 `A = ε` 这两条生成规则，从 `A` 生成 `ααα` 这个句子时，展开顺序是 `A -> αA -> ααA -> αααA -> ααα`。

这样看来，`*`、`?` 等记法只不过是快捷方式。不过，短写法更容易理解，也更理想，所以能使用短写法时，通常会用这些记法简洁地描述。

### 简单的生成规则

作为使用 EBNF 描述语法的例子，请考虑下面的生成规则。

```ebnf
expr = num ("+" num | "-" num)*
```

这里假设 `num` 已在别处被定义为表示数值的符号。在这个语法中，`expr` 表示先有一个 `num`，之后有 0 个以上的“`+` 与 `num`，或者 `-` 与 `num`”。这条规则实际上表示加减法表达式的语法。

从 `expr` 出发展开，可以生成任意加减法字符串，例如 `1`、`10+5`、`42-30+2`。请看下面的展开结果。

```text
expr -> num -> "1"

expr -> num "+" num
     -> "10" "+" "5"

expr -> num "-" num "+" num
     -> "42" "-" "30" "+" "2"
```

这种展开过程不仅可以用箭头按展开顺序表示，也可以用树结构表示。上面表达式的语法树如下。

```text
expr
└── num
    └── 1

expr
├── num
│   └── 10
├── +
└── num
    └── 5

expr
├── num
│   └── 42
├── -
├── num
│   └── 30
├── +
└── num
    └── 2
```

用树结构表示后，哪个非终结符展开成了哪些符号就变得容易理解。

像上图这样，包含输入中所有 token、与语法完全一一匹配的语法树，有时称为“具体语法树”（concrete syntax tree）。这个术语常在需要与抽象语法树对比时使用。

不过，在上面的具体语法树中，“加减法从左到右计算”这个规则并没有通过树的形状表示出来。在这里说明的语法中，这类规则不是用 EBNF 表示，而是会在语言规格书中用文字附注写明“加减法从左侧开始计算”。parser 会同时考虑 EBNF 和附注，读取表示表达式的 token 序列，并构建能恰当表示表达式求值顺序的抽象语法树。

因此，在上述语法中，EBNF 表示的具体语法树与 parser 输出的抽象语法树形状只是在大致上相符。也可以定义一种语法，让抽象语法树和具体语法树尽量具有相同结构；但那样语法会变得冗长，也会让人难以理解 parser 应该怎么写。上面这种语法是在形式语法描述的严密性与自然语言补充说明的易懂性之间取得平衡的、易于处理的语法表达方法。

### 用生成规则表示运算符优先级

生成规则是表达语法的非常强大的工具。只要对语法做一些设计，运算符优先级也可以在生成规则中表示。该语法如下。

```ebnf
expr = mul ("+" mul | "-" mul)*
mul = num ("*" num | "/" num)*
```

以前的规则中，`expr` 会直接展开为 `num`；这次的规则中，`expr` 会经由 `mul` 展开到 `num`。`mul` 是乘除法的生成规则，而执行加减法的 `expr` 可以说把 `mul` 当作一个部件来使用。在这个语法中，乘除法先结合这一规则会自然地在语法树中表达出来。下面具体看几个例子。

```text
1*2+3

expr
├── mul
│   ├── num 1
│   ├── *
│   └── num 2
├── +
└── mul
    └── num 3
```

```text
1+2*3

expr
├── mul
│   └── num 1
├── +
└── mul
    ├── num 2
    ├── *
    └── num 3
```

```text
1*2+3*4*5

expr
├── mul
│   ├── num 1
│   ├── *
│   └── num 2
├── +
└── mul
    ├── num 3
    ├── *
    ├── num 4
    ├── *
    └── num 5
```

在上面的树结构中，乘法总是比加法更靠近树的末端方向。实际上，由于没有从 `mul` 回到 `expr` 的规则，所以根本构造不出“乘法下面有加法”的树；尽管如此，仅凭这样简单的规则就能把优先级很好地表示为树结构，仍然让人觉得相当神奇。读者也请实际对照生成规则和语法树，确认语法树是正确的。

### 包含递归的生成规则

在生成语法中，也可以正常书写递归语法。下面是给四则运算添加优先级括号后的语法生成规则。

```ebnf
expr    = mul ("+" mul | "-" mul)*
mul     = primary ("*" primary | "/" primary)*
primary = num | "(" expr ")"
```

把上述语法与之前的语法相比，可以看到过去允许 `num` 出现的地方，现在允许出现 `primary`，也就是 `num` 或 `"(" expr ")"`。换言之，在这个新语法中，被圆括号包围的表达式会以与过去单个数字相同的“结合强度”处理。来看一个例子。

下面的树是 `1*2` 的语法树。

```text
expr
└── mul
    ├── primary
    │   └── num 1
    ├── *
    └── primary
        └── num 2
```

下面的树是 `1*(2+3)` 的语法树。

```text
expr
└── mul
    ├── primary
    │   └── num 1
    ├── *
    └── primary
        ├── (
        ├── expr
        │   ├── mul
        │   │   └── primary
        │   │       └── num 2
        │   ├── +
        │   └── mul
        │       └── primary
        │           └── num 3
        └── )
```

比较这两棵树可以看到，只有 `mul` 右分支中的 `primary` 展开结果不同。出现在展开结果末端的 `primary`，既可以展开成一个数字，也可以展开成用括号括起来的任意表达式；这个规则被树结构正确反映出来。用如此简单的生成规则就能处理括号优先级，不是有点令人感动吗？

### 递归下降语法分析

如果给定 C 语言的生成规则，不断展开它，就能机械地生成从生成规则角度看正确的任意 C 程序。但是在 `9cc` 中，我们想做的恰恰相反。外部会以字符串形式给出 C 程序，我们想知道展开后会变成输入字符串的展开过程，也就是会变成与输入相同字符串的语法树结构。

实际上，对于某些生成规则，只要给出规则，就可以机械地写出代码，用来求出与该规则生成的句子相匹配的语法树。这里说明的“递归下降语法分析法”就是这类技巧之一。

以四则运算语法为例。这里再次列出四则运算的语法。

```ebnf
expr    = mul ("+" mul | "-" mul)*
mul     = primary ("*" primary | "/" primary)*
primary = num | "(" expr ")"
```

用递归下降语法分析法编写 parser 时，基本策略是把这些非终结符逐个直接映射为函数。因此 parser 会拥有 `expr`、`mul`、`primary` 这 3 个函数。每个函数都会 parse 与其名字对应的 token 序列。

具体用代码来思考。传给 parser 的输入是 token 序列。我们希望 parser 创建并返回抽象语法树，所以先定义抽象语法树节点的类型。节点类型如下。

```c
// 抽象语法树节点的种类
typedef enum {
  ND_ADD, // +
  ND_SUB, // -
  ND_MUL, // *
  ND_DIV, // /
  ND_NUM, // 整数
} NodeKind;

typedef struct Node Node;

// 抽象语法树节点的类型
struct Node {
  NodeKind kind; // 节点类型
  Node *lhs;     // 左辺
  Node *rhs;     // 右辺
  int val;       // 仅在 kind 为 ND_NUM 时使用
};
```

`lhs` 和 `rhs` 分别是 left-hand side 与 right-hand side，也就是左边和右边的意思。

也先定义创建新节点的函数。在这个语法的四则运算中，有接收左边和右边的二元运算符，以及数值这两类节点，所以准备两个对应的函数。

```c
Node *new_node(NodeKind kind, Node *lhs, Node *rhs) {
  Node *node = calloc(1, sizeof(Node));
  node->kind = kind;
  node->lhs = lhs;
  node->rhs = rhs;
  return node;
}

Node *new_node_num(int val) {
  Node *node = calloc(1, sizeof(Node));
  node->kind = ND_NUM;
  node->val = val;
  return node;
}
```

接下来，用这些函数和数据类型编写 parser。`+` 和 `-` 是左结合运算符。parse 左结合运算符的函数，可以按下面这种模式编写。

```c
Node *expr() {
  Node *node = mul();

  for (;;) {
    if (consume('+'))
      node = new_node(ND_ADD, node, mul());
    else if (consume('-'))
      node = new_node(ND_SUB, node, mul());
    else
      return node;
  }
}
```

`consume` 是之前步骤中定义的函数：当输入 stream 的下一个 token 与参数匹配时，它会向前读取 1 个 token 并返回真。

请仔细阅读 `expr` 函数。你应该能看出，`expr = mul ("+" mul | "-" mul)*` 这条生成规则被直接映射成了函数调用和循环。上面的 `expr` 函数返回的抽象语法树中，运算符是左结合的，也就是说返回节点的左侧分支会更深。

也来定义 `expr` 函数使用的 `mul` 函数。`*` 和 `/` 也是左结合运算符，所以可以用同样的模式书写。函数如下。

```c
Node *mul() {
  Node *node = primary();

  for (;;) {
    if (consume('*'))
      node = new_node(ND_MUL, node, primary());
    else if (consume('/'))
      node = new_node(ND_DIV, node, primary());
    else
      return node;
  }
}
```

上述代码的函数调用关系，直接对应 `mul = primary ("*" primary | "/" primary)*` 这条生成规则。

最后定义 `primary` 函数。`primary` 读取的不是左结合运算符，所以不会变成上述模式的代码；但只要把 `primary = "(" expr ")" | num` 这条生成规则直接对应到函数调用，就可以像下面这样书写 `primary` 函数。

```c
Node *primary() {
  // 如果下一个 token 是 "("，则应为 "(" expr ")"
  if (consume('(')) {
    Node *node = expr();
    expect(')');
    return node;
  }

  // 否则应为数值
  return new_node_num(expect_number());
}
```

这样所有函数都齐了，但它们真的能 parse token 序列吗？乍看也许不太清楚，但使用这组函数确实可以正确 parse token 序列。以表达式 `1+2*3` 为例。

最先被调用的是 `expr`。也就是说，我们假定整个表达式就是 `expr`（在这个例子中实际也是如此），然后开始读取输入。于是会发生 `expr -> mul -> primary` 这样的函数调用，读取 `1` 这个 token，并向 `expr` 返回表示 `1` 的语法树作为返回值。

接着，`expr` 中的 `consume('+')` 表达式为真，所以 `+` 这个 token 被消费，并再次调用 `mul`。此时输入剩余部分是 `2*3`。

和前面一样，`mul` 会调用 `primary`，读取 `2` 这个 token，但这次 `mul` 不会马上返回。因为 `mul` 中的 `consume('*')` 表达式为真，所以 `mul` 再次调用 `primary`，读取 `3` 这个 token。结果，`mul` 会返回表示 `2*3` 的语法树。

返回到 `expr` 后，表示 `1` 的语法树和表示 `2*3` 的语法树会被组合起来，构建出表示 `1+2*3` 的语法树，并作为 `expr` 的返回值。也就是说，`1+2*3` 被正确 parse 了。

如果把函数调用关系以及各函数读取的 token 画成图，会如下所示。下图中有对应整个 `1+2*3` 的 `expr` 层，它表示读取整个输入的 `expr` 调用。`expr` 上方有两个 `mul`，它们表示读取 `1` 和 `2*3` 的两个不同 `mul` 调用。

```text
tokens:   1       +       2       *       3
          |               |               |
       primary         primary         primary
          |               \______________/
         mul                    mul
          \______________________/
                    expr
```

下面给出稍微复杂一些的例子。下图表示 parse `1*2+(3+4)` 时的函数调用关系。

```text
tokens:   1       *       2       +       (       3       +       4       )
          |               |                       |               |
       primary         primary                 primary         primary
          \______________/                         |               |
              mul                                mul             mul
                                                   \_____________/
                                                        expr
                                                    primary
                                                       |
                                                      mul
          \____________________________________________/
                             expr
```

如果程序员不习惯递归，可能会觉得上面这样的递归函数难以理解。坦白说，即便是本应非常习惯递归的作者，也觉得这种代码能工作像某种魔法。递归代码即使理解了机制，也总会让人觉得有些不可思议，这大概就是递归的特点。请多次在脑中仔细跟踪代码，确认代码确实会正常工作。

像上面这样，把一条生成规则映射成一个函数的语法分析方法，称为“递归下降语法分析”。上述 parser 只向前看 1 个 token，决定调用哪个函数或者是否返回；这种只向前看 1 个 token 的递归下降 parser 称为 LL(1) parser。能够编写 LL(1) parser 的语法称为 LL(1) 语法。

## 栈机

前面说明了把 token 序列转换为抽象语法树的算法。通过选择考虑运算符优先级的语法，我们已经能够构造出这样的抽象语法树：与 `+`、`-` 相比，`*` 和 `/` 总是出现在更靠近分支末端的位置。那么，应该如何把这棵树转换为汇编呢？本节说明这个方法。

先思考为什么不能用与加减法相同的方法转换为汇编。在能处理加减法的编译器中，我们把 RAX 作为结果寄存器，并对它进行加法或减法。也就是说，编译出的程序只保存一个中间计算结果。

但是，如果包含乘除法，中间计算结果不一定只有一个。以 `2*3+4*5` 为例。为了执行加法，左右两边必须都已经计算完成，所以加法之前需要先计算 `2*3` 和 `4*5`。也就是说，在这个例子中，如果不能保存两个中间计算结果，就无法完成整体计算。

能简单处理这类计算的计算机称为“栈机”。这里先暂时离开 parser 创建的抽象语法树，学习栈机。

### 栈机的概念

栈机是把栈作为数据保存区域的计算机。因此，在栈机中，“push 到栈”和“从栈 pop”这两种操作是基本操作。push 会把新元素放到栈顶。pop 会从栈顶移除元素。

栈机中的运算指令作用于栈顶元素。例如，栈机的 `ADD` 指令会从栈顶 pop 两个元素，把它们相加，并把结果 push 到栈上（为了避免与 x86-64 指令混淆，虚拟栈机指令全部用大写表示）。换句话说，`ADD` 是把栈顶的两个元素替换为一个“它们相加结果”的指令。

`SUB`、`MUL`、`DIV` 指令与 `ADD` 类似，都是把栈顶的两个元素替换成它们相减、相乘、相除后的一个元素。

`PUSH` 指令把参数元素放到栈顶。这里不会使用，但也可以设想一个名为 `POP` 的指令，它从栈顶取出一个元素并丢弃。

接下来考虑用这些指令计算 `2*3+4*5`。使用上面定义的栈机，应该可以用下面这样的代码计算 `2*3+4*5`。

```text
// 计算 2*3
PUSH 2
PUSH 3
MUL

// 计算 4*5
PUSH 4
PUSH 5
MUL

// 计算 2*3 + 4*5
ADD
```

稍微详细看一下这段代码。假设栈中预先有某些值。这里这些值并不重要，所以用“...”表示。图中栈从上向下增长。

最开始两个 `PUSH` 会把 2 和 3 push 到栈上，所以紧接着执行 `MUL` 时，栈状态如下。

```text
...
2
3
```

`MUL` 会移除栈顶的两个值，也就是 3 和 2，并把它们相乘的结果，也就是 6，push 到栈上。因此执行 `MUL` 后，栈状态如下。

```text
...
6
```

接着 `PUSH` 会 push 4 和 5，所以第二个 `MUL` 执行前，栈应该如下。

```text
...
6
4
5
```

此时执行 `MUL`，5 和 4 会被移除，并替换成它们相乘的结果 20。因此执行 `MUL` 后如下。

```text
...
6
20
```

请注意，`2*3` 和 `4*5` 的计算结果已经顺利进入栈中。在这个状态下执行 `ADD`，就会计算 `20+6`，并把结果 push 到栈上，所以最终栈应该变成下面的状态。

```text
...
26
```

如果约定栈机的计算结果就是栈顶留下的值，那么 26 是 `2*3+4*5` 的结果，所以这个表达式被正确计算出来了。

在栈机中，不限于这个表达式，任何拥有多个中间结果的表达式都可以计算。使用栈机时，只要任意子表达式都遵守“执行后把一个元素作为结果留在栈上”这一约定，就可以用上述方法顺利编译。

### 专栏：CISC 与 RISC

x86-64 是从 1978 年发布的 8086 渐进发展而来的指令集，是典型的被称为“CISC”（complex instruction set computer）的处理器风格。CISC 处理器的特点包括：机器码运算不仅可以接收寄存器，也可以接收内存地址；机器码指令长度可变；拥有许多对汇编程序员方便的、能用一条指令完成复杂操作的指令。

与 CISC 相对，1980 年代发明了“RISC”（reduced instruction set computer）。RISC 处理器的特点包括：运算一定只在寄存器之间进行，对内存的操作只有加载到寄存器和从寄存器存储；所有机器码指令长度相同；不拥有对汇编程序员方便的复合指令，只具备编译器生成的简单指令。

x86-64 是少数幸存的 CISC 之一；除 x86-64 以外，主要处理器几乎全部基于 RISC。具体来说，ARM、PowerPC、SPARC、MIPS、RISC-V 等都是 RISC 处理器。

RISC 中没有 x86-64 这样在内存与寄存器之间直接进行的运算。也没有寄存器别名。也没有“某个特定整数寄存器会在某条特定指令中被特殊使用”这样的规则。从这类指令集成为主流的现代视角看，x86-64 指令集显得比较古老。

RISC 处理器因为设计简单而容易高速化，并席卷了处理器行业。那么为什么 x86-64 成功存活了下来呢？这里有市场对高速 x86 处理器的巨大需求，因为它能利用既有软件资产；也有 Intel 以及 Intel 兼容芯片厂商为回应这种需求而进行的技术创新。Intel 在 CPU 的指令 decoder 中把 x86 指令内部转换为某种 RISC 指令，把 x86 在内部变成了 RISC 处理器。这样一来，RISC 成功实现高速化所用的同类技术也可以应用到 x86 上。

### 编译到栈机

本节说明把抽象语法树转换为栈机代码的方法。做到这一点后，就可以 parse 由四则运算组成的表达式、组装抽象语法树，并把它编译到使用 x86-64 指令实现的栈机上运行。也就是说，我们就能写出可以处理四则运算的编译器。

在栈机中，计算子表达式后，无论它是什么，其结果值都会留在栈顶。请考虑下面这样的树。

```text
    +
   / \
  A   B
```

`A` 和 `B` 是对部分树的抽象表示，实际意味着某种类型的节点。但是在编译整棵树时，它们具体的类型和树形并不重要。编译这棵树时，只需要这样做。

1. 编译左子树
2. 编译右子树
3. 输出把栈中两个值替换为它们相加结果的代码

执行第 1 步的代码后，无论具体代码是什么，表示左子树结果的一个值都应该位于栈顶。同样，执行第 2 步的代码后，表示右子树结果的一个值也应该位于栈顶。因此，为了计算整棵树的值，只要把这两个值替换为它们的合计值即可。

像这样，把抽象语法树编译到栈机时，需要递归地思考，一边沿着树向下走，一边不断输出汇编。对于不习惯递归思路的读者，这可能稍显困难；但处理树这种自相似数据结构时，递归是标准技巧。

用下面的例子具体思考。

```text
      +
     / \
    2   *
       / \
      3   4
```

进行代码生成的函数接收树的根节点。

按照上述步骤，这个函数最先做的是编译左子树，也就是编译数值 2。计算 2 的结果仍然是 2，所以该子树的编译结果是 `PUSH 2`。

接下来，代码生成函数会尝试编译右子树。于是会递归地编译该子树的左侧，结果输出 `PUSH 3`。然后编译该子树的右侧，输出 `PUSH 4`。

之后，代码生成函数会在从递归调用返回的过程中，根据子树运算符的类型输出代码。最先输出的是把栈顶两个元素替换成它们相乘结果的代码。然后输出把栈顶两个元素替换成它们相加结果的代码。结果会输出下面这样的汇编。

```text
PUSH 2
PUSH 3
PUSH 4
MUL
ADD
```

使用这种方法，就能机械地把抽象语法树落到汇编中。

### 在 x86-64 中实现栈机的方法

到目前为止讲的是虚拟栈机。实际的 x86-64 不是栈机，而是寄存器机。x86-64 的运算通常定义在两个寄存器之间，并不是定义为作用于栈顶两个值。因此，为了在 x86-64 中使用栈机技巧，需要在寄存器机上以某种意义模拟栈机。

用寄存器机模拟栈机相对简单。只需要用多条指令实现栈机中作为一条指令的操作。

下面说明具体方法。

首先准备一个指向栈顶元素的寄存器。这个寄存器称为栈指针。如果想 pop 栈顶两个值，就取出栈指针指向的两个元素，并按取出的元素数量修改栈指针。同样，push 时只要一边修改栈指针的值，一边向它指向的内存区域写入即可。

x86-64 的 RSP 寄存器就是按照作为栈指针使用来设计的。x86-64 的 `push`、`pop` 等指令，会隐式地把 RSP 用作栈指针，在修改其值的同时访问 RSP 指向的内存。因此，把 x86-64 指令集当作栈机使用时，自然应把 RSP 用作栈指针。现在马上把 `1+2` 这个表达式当作在 x86-64 栈机上运行来编译。下面是 x86-64 汇编。

```asm
// push 左边和右边
push 1
push 2

// 将左边和右边 pop 到 RAX 与 RDI 并相加
pop rdi
pop rax
add rax, rdi

// 将相加结果 push 到栈
push rax
```

x86-64 没有“把 RSP 指向的两个元素相加”这样的指令，所以需要先加载到寄存器中执行加法，再把结果重新 push 到栈上。上面的 `add` 指令做的就是这个操作。

同样，用 x86-64 实现 `2*3+4*5` 会如下所示。

```asm
// 计算 2*3 并将结果 push 到栈
push 2
push 3

pop rdi
pop rax
mul rax, rdi
push rax

// 计算 4*5 并将结果 push 到栈
push 4
push 5

pop rdi
pop rax
mul rax, rdi
push rax

// 将 stack top 的两个值相加
// 也就是计算 2*3+4*5
pop rdi
pop rax
add rax, rdi
push rax
```

像这样，使用 x86-64 的栈操作指令，即使在 x86-64 上，也能运行相当接近栈机的代码。

下面的 `gen` 函数就是把这种方法直接实现为 C 函数。

```c
void gen(Node *node) {
  if (node->kind == ND_NUM) {
    printf(" push %d\n", node->val);
    return;
  }

  gen(node->lhs);
  gen(node->rhs);

  printf("     pop rdi\n");
  printf("     pop rax\n");

  switch (node->kind) {
  case ND_ADD:
    printf(" add rax, rdi\n");
    break;
  case ND_SUB:
    printf(" sub rax, rdi\n");
    break;
  case ND_MUL:
    printf(" imul rax, rdi\n");
    break;
  case ND_DIV:
    printf(" cqo\n");
    printf(" idiv rdi\n");
    break;
  }

  printf("     push rax\n");
}
```

虽然这并不是 parse 或代码生成中的关键点，但上面代码使用了规格有些 tricky 的 `idiv` 指令，所以这里说明一下。

`idiv` 是执行有符号除法的指令。如果 x86-64 的 `idiv` 规格更直观，那么上面的代码本来会想写成 `idiv rax, rdi`。但是 x86-64 中不存在这种接收两个寄存器的除法指令。取而代之，`idiv` 会隐式地取 RDX 和 RAX，把它们合起来视为一个 128 位整数，再除以参数寄存器中的 64 位值，把商设置到 RAX，把余数设置到 RDX。使用 `cqo` 指令，可以把 RAX 中的 64 位值扩展为 128 位并设置到 RDX 和 RAX，所以在上面的代码中，调用 `idiv` 前先调用了 `cqo`。

到这里，栈机的说明就结束了。读到这里，读者应该已经能够进行复杂语法分析，并把语法分析结果得到的抽象语法树落到机器码中。为了运用这些知识，回到编译器制作工作中吧！

### 专栏：优化编译器

本章中作者用于说明的 x86-64 汇编可能看起来相当低效。例如，把数值 push 到栈上再 pop 出来的指令，如果直接写成把该值 `mov` 到寄存器的指令，本应一条指令就够了。读者中可能有人已经产生了想从这类汇编中去掉冗余并进行优化的冲动。但是，请不要输给这种诱惑。在最初的代码生成中，优先考虑编译器实现的容易程度，并输出冗余代码，是合理的。

如果需要，之后可以给 `9cc` 添加优化 pass。再次扫描生成的汇编，把以特定模式出现的指令序列替换为其他指令序列并不困难。例如，制定“`push` 之后紧跟的 `pop` 替换为 `mov`”，或者“连续的 `add` 如果都是把立即数加到同一个寄存器，就替换为一个加上这些立即数合计值的 `add`”这样的规则，并机械地应用，就可以在不改变含义的情况下，把冗余代码替换为更高效的代码。

### 专栏：9cc 中的内存管理

读到这里，读者可能会疑惑这个编译器中的内存管理是怎样的。到目前为止出现的代码中，虽然使用了 `calloc`（`malloc` 的一种变体），但没有调用 `free`。也就是说，分配的内存没有被释放。这是不是太偷懒了？

实际上，这种“把不做内存管理作为内存管理策略”的设计，是作者在考虑各种权衡后有意选择的设计。

这个设计的优点是，由于不释放内存，可以像使用带垃圾回收的语言一样写代码。这样不仅不需要编写进行内存管理的代码，还能从根本上切断与手动内存管理相关的难解 bug。

另一方面，如果考虑在普通 PC 这样的计算机上运行，不调用 `free` 导致的问题实际上并不多。编译器是一个短命程序，只读取一个 C 文件并输出汇编。程序结束时仍然分配着的内存会由 OS 全部自动释放。因此，问题只剩下总共会分配多少内存；根据作者实际测量，即使编译相当大的 C 文件，内存使用量也只有约 100MiB。因此，不 `free` 是现实中有效的策略。例如 D 语言编译器 DMD 也基于同样想法，采用只 `malloc` 而不 `free` 的策略。

## 步骤 6：单目加号和单目减号

执行减法的 `-` 运算符，不仅可以像 `5-3` 这样写在两个项之间，也可以像 `-3` 这样写在单独项之前。同样，`+` 运算符也可以省略左边，写成 `+3`。这种只接收一个项的运算符称为“单目运算符”（unary operator）。与此相对，接收两个项的运算符称为“二元运算符”（binary operator）。

C 中除了 `+` 和 `-` 以外，还有获取指针的 `&`、解引用指针的 `*` 等单目运算符；但这个步骤中只实现 `+` 和 `-`。

单目 `+` 和单目 `-` 与二元 `+`、`-` 使用相同符号，但定义不同。二元 `-` 被定义为从左边减去右边的运算；但单目 `-` 本来就没有左边，因此二元 `-` 的定义不能直接使用。在 C 中，单目 `-` 被定义为反转右边正负号的运算。单目 `+` 是直接返回右边的运算符。这个运算符即使没有也没关系，但因为存在单目 `-`，所以顺带存在。

更合适的理解是，`+` 和 `-` 存在多个同名运算符，它们分别是单目和二元，定义相似但不同。到底是单目还是二元，需要根据上下文区分。包含单目 `+/-` 的新语法如下。

```ebnf
expr    = mul ("+" mul | "-" mul)*
mul     = unary ("*" unary | "/" unary)*
unary   = ("+" | "-")? primary
primary = num | "(" expr ")"
```

上述新语法增加了名为 `unary` 的新非终结符，并且 `mul` 不再使用 `primary`，而是使用 `unary`。`X?` 是 EBNF 语法，表示 optional，也就是 `X` 出现 0 次或 1 次。在 `unary = ("+" | "-")? primary` 这条规则中，非终结符 `unary` 表示可以有也可以没有一个 `+` 或 `-`，之后跟着 `primary`。

请确认 `-3`、`-(3+5)`、`-3*+5` 等表达式都匹配这个新语法。下面展示 `-3*+5` 的语法树。

```text
expr
└── mul
    ├── unary
    │   ├── -
    │   └── primary
    │       └── num 3
    ├── *
    └── unary
        ├── +
        └── primary
            └── num 5
```

让我们修改 parser，使其符合这个新语法。和以往一样，把语法直接映射为函数调用，parser 的修改就应该完成了。parse `unary` 的函数如下。

```c
Node *unary() {
  if (consume('+'))
    return primary();
  if (consume('-'))
    return new_node(ND_SUB, new_node_num(0), primary());
  return primary();
}
```

这里在 parse 阶段就把 `+x` 替换成 `x`，把 `-x` 替换成 `0-x`。因此这个步骤中不需要修改代码生成器。

写几个测试，并和添加单目 `+/-` 的代码一起 check in，这个步骤就完成了。写测试时，请让测试结果落在 0 到 255 范围内。像 `-10+20` 这样的表达式使用了单目 `-`，但整体值仍为正数，可以使用这类测试。

### 专栏：单目加号与语法好坏

单目 `+` 运算符在最初的 C 编译器中并不存在，而是在 1989 年 ANSI（美国国家标准协会）标准化 C 时正式加入语言的。既然有单目 `-`，那么也有单目 `+` 会让对称性更高，从这个意义上说确实不错；但实际上单目 `+` 并没有特别用途。

另一方面，把单目 `+` 加入语法也有副作用。假设不熟悉 C 的人把 `+=` 运算符误写成 `i =+ 3`。如果没有单目 `+`，这只是非法表达式；但因为有单目 `+`，它会被解释为与 `i = +3` 相同，并作为“给 `i` 赋值 3”的合法赋值表达式被编译器默默接受。

ANSI 的 C 语言标准化委员会是在理解上述问题的基础上，仍然做出了把单目 `+` 加入语言的判断。读者怎么看？如果你当时属于 C 标准化委员会，会赞成还是反对？

## 步骤 7：比较运算符

本节实现 `<`、`<=`、`>`、`>=`、`==`、`!=`。这些比较运算符看起来似乎有特殊含义，但实际上和 `+`、`-` 等一样，都是接收两个整数并返回一个整数的普通二元运算符。就像 `+` 返回两边相加的结果一样，例如 `==` 在两边相同时返回 1，不同时返回 0。

### 修改 tokenizer

到目前为止处理的符号 token 长度都是 1 个字符，代码中也以此为前提；但为了处理 `==` 等比较运算符，需要泛化代码。为了能把字符串长度保存到 token 中，给 `Token` 结构体保存一个名为 `len` 的成员。新的结构体类型如下。

```c
struct Token {
  TokenKind kind; // token 类型
  Token *next;   // 下一个输入 token
  int val;       // kind 为 TK_NUM 时的数值
  char *str;     // token 字符串
  int len;       // token 长度
};
```

伴随这个修改，也需要修改 `consume`、`expect` 等函数，让这些函数接收字符串而不是字符。下面给出修改后的例子。

```c
bool consume(char *op) {
  if (token->kind != TK_RESERVED ||
      strlen(op) != token->len ||
      memcmp(token->str, op, token->len))
    return false;
  token = token->next;
  return true;
}
```

tokenize 由多个字符构成的符号时，必须先 tokenize 长 token。例如，如果剩余字符串以 `>` 开头，但没有先用 `strncmp(p, ">=", 2)` 这样检查它是否可能是 `>=`，而是先检查它是否以 `>` 开头，那么 `>=` 就会被错误地 tokenize 成 `>` 和 `=` 两个 token。

### 新语法

为了给 parser 添加比较运算符支持，来思考加入比较运算符后的语法会变成什么样。把到目前为止出现的运算符按优先级从低到高排列，如下所示。

1. `==` `!=`
2. `<` `<=` `>` `>=`
3. `+` `-`
4. `*` `/`
5. 单目 `+` 单目 `-`
6. `()`

优先级可以用生成语法表示，优先级不同的运算符会映射到不同的非终结符。像 `expr`、`mul` 那样思考语法，加入比较运算符后的新语法如下。

```ebnf
expr       = equality
equality   = relational ("==" relational | "!=" relational)*
relational = add ("<" add | "<=" add | ">" add | ">=" add)*
add        = mul ("+" mul | "-" mul)*
mul        = unary ("*" unary | "/" unary)*
unary      = ("+" | "-")? primary
primary    = num | "(" expr ")"
```

`equality` 表示 `==` 和 `!=`，`relational` 表示 `<`、`<=`、`>`、`>=`。这些非终结符可以使用 parse 左结合运算符的模式，直接映射为函数。

另外，在上述语法中，为了表示整个表达式是 `equality`，把 `expr` 和 `equality` 分离了。也可以把 `equality` 的右边直接写到 `expr` 的右边，但上面这种语法大概更容易看。

### 专栏：简单冗长的代码与高级简洁的代码

在递归下降语法分析中，代码几乎直接对应生成规则，因此 parse 相似规则的函数会有相似外观。到目前为止写的 `relational`、`equality`、`add`、`mul`，应该也都是外观相似的函数。

自然会想到：能否用 C 宏、C++ template、高阶函数或代码生成等元编程技巧，把这类函数共有的模式很好地抽象出来？实际上，确实可以这么做。但是本书有意没有这样做。理由如下。

简单代码即使稍微冗长，也容易理解。即使之后要给相似函数添加相似修改，实际上也并不费事。另一方面，高度抽象的代码需要先理解抽象机制，再理解它是如何被使用的，因此容易变得难懂。例如，如果本书从“使用元编程编写生成递归下降语法分析函数的函数”开始讲解，这本书就会变得难得多。

并不需要总是追求写出技巧精巧的简洁代码。如果总是以此为目标，往往会把代码复杂化到不能再复杂的程度。

写代码的人会成为那段代码的专家，所以容易从专家视角认为简洁、无浪费的代码就是好代码。但大多数代码读者并不共享作者的感觉，也没有必要熟悉到那个程度，因此作为代码作者，需要在一定程度上怀疑自己的感觉。根据需要有意写下“似乎还有更好写法的简单代码”，是制作易理解、易维护程序的一项重要技巧。

### 生成汇编代码

在 x86-64 中，比较使用 `cmp` 指令完成。从栈中 pop 两个整数并进行比较，如果相同则把 1 设置到 RAX，否则把 0 设置到 RAX 的代码如下。

```asm
pop rdi
pop rax
cmp rax, rdi
sete al
movzb rax, al
```

这段汇编虽然很短，但内容不少，所以一步步来看。

最开始两行从栈中 pop 值。第 3 行比较（compare）这些 pop 出来的值。比较结果去哪里了呢？在 x86-64 中，比较指令的结果会被设置到一个特殊的“标志寄存器”。标志寄存器会在每次执行整数运算或比较运算指令时更新，它包含结果是否为 0、是否发生溢出、结果是否小于 0 等 bit。

标志寄存器不是普通整数寄存器，所以如果想把比较结果设置到 RAX，需要把标志寄存器中的特定位复制到 RAX。执行这个操作的是 `sete` 指令。`sete` 指令在前一个 `cmp` 指令检查的两个寄存器值相等时，把 1 设置到指定寄存器（这里是 AL）；否则设置 0。

`AL` 是本书到目前为止还没出现过的新寄存器名，但实际上 `AL` 只是指向 RAX 低 8 bit 的别名寄存器。因此，`sete` 给 AL 设置值时，RAX 也会自动更新。不过，通过 AL 更新 RAX 时，高 56 bit 会保持原值，所以如果想把整个 RAX 设置为 0 或 1，需要把高 56 bit 清零。执行这一点的是 `movzb` 指令。如果 `sete` 能直接写入 RAX 就好了，但 `sete` 的规格是只能接收 8 bit 寄存器作为参数，所以比较指令中会这样使用两条指令把值设置到 RAX。

```text
RAX: [ high 56 bits ][ AL: low 8 bits ]
```

使用其他指令代替 `sete`，就可以实现其他比较运算符。`<` 使用 `setl`，`<=` 使用 `setle`，`!=` 使用 `setne`。

`>` 和 `>=` 不需要在代码生成器中支持。请在 parser 中交换左右两边，把它们改读为 `<` 或 `<=`。

### 专栏：标志寄存器与硬件

x86-64 这种“值的比较结果会隐式保存到不同于普通整数寄存器的特殊寄存器中”的规格，刚开始可能让人觉得难懂。实际上，在 RISC 处理器中，有些指令集不喜欢拥有标志寄存器，而是把值的比较结果设置到普通寄存器中。例如 RISC-V 就是这样的指令集。

但是，从硬件实现的角度看，如果是朴素实现，制作标志寄存器非常简单。也就是说，执行整数运算时，把结果的线路分支出来连接到另一组逻辑，让那边检查结果是否为零（所有线是否为 0）、结果是否为负（最高位 bit 的线是否为 1）等，并把结果设置到标志寄存器的各个 bit 中即可。拥有标志寄存器的 CPU 正是这样实现的，每次执行整数运算时，标志寄存器也会顺便更新。

在这种机制中，不只是 `cmp`，`add`、`sub` 等也会更新标志寄存器。实际上，`cmp` 的本质是一种只更新标志寄存器的特殊 `sub` 指令。如果执行 `sub rax, rdi`，然后查看标志寄存器，就能知道 RAX 和 RDI 的大小关系；但这样 RAX 会被更新，所以准备了不写入整数寄存器的 `sub`，也就是 `cmp`。

在软件中，“顺便计算某些东西”一定会花费额外时间；但在硬件中，分出线路并额外使用晶体管本身不会产生时间惩罚，所以在朴素硬件实现中，每次更新标志寄存器并不存在时间成本。
